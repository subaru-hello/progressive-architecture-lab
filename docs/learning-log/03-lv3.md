# Lv3 — Kubernetes 複数ノード + HPA（自動スケール）

記録日: 2026-07-06 / マシン: 8 コア / k3d 5.9.0（3 ノード）/ k3s v1.35.5

## やったこと
- k3d を 3 ノード（1 server + 2 agents）で作り直し。
- HPA(CPU 50%, min2/max10) + topologySpreadConstraints を追加。
- 持続 CPU 負荷をかけ、**自動スケール 2→10** と **エラー 0%**（Lv2 の 98.8% 事故の解消）を実測。

## 負荷 → 自動スケール（実況）
```console
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8081 -e CPU=2 \
    grafana/k6 run --vus 12 --duration 150s - < load/cpu.js &
# ポーリング
t+15s  | 11%/50%   reps=2  | running=2
t+30s  | 126%/50%  reps=2  | running=5      # スケール開始
t+45s  | 485%/50%  reps=6  | running=10
t+60s  | 328%/50%  reps=10 | running=10     # 上限到達
t+120s | 167%/50%  reps=10 | running=10
（負荷終了+45s） 25%/50% reps=10→5           # 縮小開始（60s 安定化後）
```
- **負荷開始 ~45 秒で 2 → 10 Pod に自動スケール。** 終了後は縮小。上げ即・下げ慎重の挙動どおり。

```console
$ kubectl -n pal get pods -l app=api -o custom-columns=NODE:.spec.nodeName --no-headers | sort | uniq -c
   3 k3d-pal-agent-0
   4 k3d-pal-agent-1
   3 k3d-pal-server-0        # 10 Pod が 3/4/3 で 3 ノードに分散
```

## 事故は起きたか（Lv2 との対比）
```console
$ grep -E 'checks_succeeded|http_req_failed|http_req_duration' /tmp/pal-load.log
    checks_succeeded...: 100.00% 3504 out of 3504
    http_req_failed....: 0.00%  0 out of 3504
    http_req_duration..: avg=513.52ms p(95)=1.71s max=13.72s
```
| | Lv2（固定 3 Pod, cpu=10） | Lv3（HPA 2→10, cpu=2） |
|---|---:|---:|
| エラー率 | **98.8%**（probe 連鎖） | **0.00%** ✓ |

- **HPA が飽和前に Pod を足したので probe 連鎖が起きなかった** = 教訓5「真の解＝飽和前に増設」を実証。
- レイテンシは高い（avg 513ms）。8 コアに 10 Pod ＋ k6 を詰めた**オーバーサブスクリプション**のため。

## ⚠️ 正直な注記
k3d の「3 ノード」は 1 台の Mac 上の 3 コンテナ＝**同じ 8 コアを共有**する。
HPA・スケジューリング・topologySpread の挙動学習には十分だが、**実ハードのコアは増えていない**。
自宅の実サーバー群なら、ノード追加＝実コア増になる（＝レイテンシも本当に改善する）。

## 気づき（LEARNINGS.md に反映済）
1. **HPA は「飽和前に増設」で probe 事故を回避できる。** 同種負荷で Lv2=98.8%エラー → Lv3=0%。
2. **HPA の分母は `requests.cpu`。** 利用率 = 使用量 / requests。metrics-server 必須。`limits.cpu` は付けない(Lv2 の教訓)。
3. **Pod は増やせてもコア／ノードに上限。** max 到達後も利用率 166% = 真の限界はノード数。次は Cluster Autoscaler / 物理ノード追加。
4. **単一 DB は水平スケールの外に残る。** api 10 Pod でも Postgres は 1。次の壁はリードレプリカ / PgBouncer。

## このラボの到達点
Lv0（1 プロセス）→ Lv1（レプリカ+LB）→ Lv2（宣言的 k8s）→ Lv3（複数ノード自動スケール）を
**同じアプリのまま基盤だけ乗せ替えて**登り切った。スケールの本質・LB・probe 事故・計測リテラシー・
宣言的運用・自動スケールを一通り手を動かして確認できた。残る本物の壁（DB・実ノード増設・worker_threads）が
次の探究テーマ。
