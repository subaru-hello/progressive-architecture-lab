# Lv14 — modular monolith：env 1 個差で「schema 結合」を runtime 無料で切る

記録日: 2026-07-12 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~4-5

## やったこと
- Lv13 と**同一トポロジ**（1 プロセス・共有 1 Postgres・同一 image）。**差は env 1 個だけ**:
  `ORDERS_CROSS_CONTEXT: join → port`。
- **port モード**: orders は items テーブルを直接触らず、**in-process の `ItemsPort`（関数呼び出し）越し**に
  在庫を引く（`itemsPort.decrementStock(itemId, qty)`）。プロセス境界は跨がないので、この呼び出しは
  ただの関数呼び出し = network も serialization も無い。
- **重要**: この `ItemsPort` interface は **Lv15 micro と同一**。Lv14 では in-process アダプタ、Lv15 では
  HTTP アダプタを挿すだけ。modular monolith は「micro の内部境界を、まだプロセスは割らずに引いた」状態。

## 環境 / セットアップ
```bash
docker compose -f stages/14-modular-monolith/docker-compose.yml up --build -d
# env: SERVICE=all, ORDERS_CROSS_CONTEXT=port, SEED_DEMO=1   ← Lv13 との差は port の 1 語のみ
```

## コマンドと出力
```console
### 疎通（in-process ItemsPort 経由）###
$ curl -s -XPOST :3000/orders -H 'x-auth-token: demo-token-2' -d '{"itemId":5,"qty":2}'
{"instance":"...","order":{"id":1,"user_id":2,"item_id":5,"qty":2,...}}
$ curl -s :3000/items/5     # stock -2（ポート越しでも在庫は正しく減る）
{"...","item":{"id":5,...,"stock":999998,...}}
POST /orders（qty=999999999） -> 409   # 在庫不足も port 経由で正しく伝搬

### steady 負荷（RATE=50, 30s, Lv13 と同条件）###
order_latency: avg=6.03ms med=5.38ms p90=7.45ms p95=8.13ms max=132ms   http_req_failed=0.00%
```

## 実測データ（Lv13 との対比）
| 指標 | Lv13 monolith(join) | Lv14 modular(port) | 差 |
|---|---|---|---|
| order_latency med | 5.34ms | **5.38ms** | +0.04ms（＝ノイズ） |
| order_latency p95 | 7.89ms | **8.13ms** | +0.24ms |
| プロセス数 / DB | 1 / 共有 1 | 1 / 共有 1 | 同一 |
| cross-context | 直 SQL JOIN | **in-process ポート** | ← ここだけ |

## いちばんの気づき — 「差が出ないこと」に意味がある
1. **latency は Lv13 と実質一致**（med 5.34 vs 5.38、差はノイズ）。当然で、port モードでも呼び出しは
   **同一プロセス内の関数呼び出し**——network も直列化も挟まらない。**modular 化は runtime コストがゼロ**。
2. **なのに構造は変わった**。orders は items テーブルの schema を**もう知らない**（`ItemsPort.decrementStock`
   という契約しか知らない）。items の内部テーブルを作り変えても、契約が同じなら orders は無傷。
   monolith(join) は items テーブルを直参照しているので同じ変更で壊れる——**この非対称が modular 化の対価回収**。
3. **これは Lv12「warm 定常は一致」と同型の語り**。数字が並ぶこと自体が結論: 「境界を引く」コストは
   runtime ではなく設計規律で払う。**支払いが起きるのはプロセスを割った瞬間（Lv15）**。

## 払ったコスト / トレードオフ
- **規律コスト（runtime ではなく人間側）**: 「orders は items テーブルを触ってはいけない・ポート越しだけ」を
  守り続ける必要がある。境界はコードレビューでしか守れない（1 プロセス内なので物理的な壁は無い）。
- **まだ独立デプロイ・独立スケール・障害隔離は得られない**（1 プロセスのまま）。それらは Lv15 で
  「network 税 + 分散データ整合の痛み」と引き換えに初めて手に入る。

## 実装メモ / 罠
- `ItemsPort`/`UsersPort` の interface は 1 つ。in-process アダプタ(`InProcessItemsAdapter`, repo 直呼び)と
  HTTP アダプタ(`HttpItemsAdapter`, fetch)を、`ITEMS_SERVICE_URL` の有無で composition root が挿し替える。
  Lv14 は URL 未設定 → in-process。**ドメインコードは Lv15 と完全に同一バイナリ**。

## 次へ（Lv15）
- **プロセスを 3 つに割り**、`ItemsPort`/`UsersPort` の in-process アダプタを **HTTP アダプタ**に差し替え、
  さらに **DB-per-service**（各サービスが自分の Postgres を所有）にする。ここで初めて:
  ①order 作成が auth+在庫の **HTTP hop** を踏む（latency 税）、②orders は items を JOIN できず
  **在庫デクリメントと order INSERT が別 DB 跨ぎ = 非トランザクション**（Lv13 でタダだった原子性を失う）、
  ③依存サービスを落とすと **blast radius が隔離される**（monolith は全滅）。分解の対価を数値で回収する。
