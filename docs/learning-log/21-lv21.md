# Lv21 — 2PC 自動 crash-recovery：in-doubt は「決定を永続化」して「起動時に再解決」すれば消える

記録日: 2026-07-15 / マシン: 8 コア Mac / Docker Compose(2DB) / 機能テスト(latency 非依存)

## やったこと
- Lv19/Lv20 で踏んだ **2PC の in-doubt** —— coordinator が「両者 prepare 済」の後、commit/rollback を出す前に死ぬと、
  両DBに prepared txn が**行ロックを握って宙吊り**になり、**手動 `ROLLBACK PREPARED`**(DBA 介入)が要った —— を、
  **古典的 2PC coordinator recovery** で自動化した。app コードのみ変更(外部 HTTP 契約は不変・内部 endpoint 1本追加)。
- **決定ジャーナル(commit point)**: coordinator DB(orders-db)に `tx_journal(gid, decision)`。両者 prepare 後に
  `decision='commit'` を**永続化してから** commit を出す。**この journal write が commit point**(＝この瞬間以降、
  クラッシュしても必ず commit に前進する)。**行が有る=commit 決定 / 行が無い=abort 決定**(＝**presumed-abort**:
  abort レコードは書かない古典的最適化)。
- **起動時リゾルバ**: 再起動時(=プロセス回復)に一度だけ走り、`pg_prepared_xacts`(orders-db ローカル + items は
  HTTP `/internal/tx/prepared`)と `tx_journal` を突合。gid が journal に有れば**両者 commit**(冪等)、無ければ**両者 rollback**(冪等)。
  **listen の前に await 完走**させる(schema 初期化後に走らせ tx_journal 未作成レースを断つ / 将来 coordinator を
  多重化しても live traffic の in-flight 2pc とレースしない)。
- **deferred-delete(correctness の核心)**: journal 行は「**両参加者の commit を確認できた時だけ**」削除。items 未達なら
  消さず次回起動に持ち越す(deferred)。先に消すと items に prepared 残存時、次回「決定なし→abort」と誤判定して
  orders(commit済) と items(rollback) が食い違う **split-brain** になる。

## コマンドと出力
```console
### DEMO 1: COMMIT 側リカバリ（FAULT_POINT=after-journal = journal 書込後・commit 前にクラッシュ）###
inject 1 order → HTTP 500
  db_prepared=1  items_prepared=1  journal=ord-da62…=commit  orders=1  item3_stock=1000000  # 宙吊り(未確定)
restart api（FAULT 解除 = プロセス再起動 = 起動時リゾルバ）
  resolver done: committed=1 aborted=0 deferred=0 scanned=1                                 # 決定=commit を発見→両者 commit
  db_prepared=0  items_prepared=0  journal=0  orders=2  item3_stock=999999                  # 確定・stock 減・journal 掃除

### DEMO 2: ABORT 側リカバリ（FAULT_POINT=after-prepare-all = journal 書込前にクラッシュ）###
inject 1 order → HTTP 500
  db_prepared=1  items_prepared=1  journal=0  orders=2  item3_stock=999999                  # 宙吊り(決定なし)
restart api
  resolver done: committed=0 aborted=1 deferred=0 scanned=1                                 # 決定なし→両者 rollback
  db_prepared=0  items_prepared=0  journal=0  orders=2  item3_stock=999999                  # rollback・stock 復旧・order 据置

### 起動ログ（リゾルバは listen の前に完走）###
"2pc resolver done" (scanned=1)  →  "Server listening at http://0.0.0.0:3000"
```

## Lv19 との対比（同じ in-doubt が手動→自動に）
| | Lv19（素の 2PC） | Lv21（+決定ジャーナル+リゾルバ） |
|---|---|---|
| coordinator が prepare 後に死ぬ | prepared 宙吊り・行ロック保持 | 同じく宙吊りになるが… |
| 回復手段 | **手動 `ROLLBACK PREPARED`**（DBA 介入・放置で後続ブロック） | **再起動で自動再解決**（commit/abort を決定通りに前進） |
| commit/abort の判断根拠 | 人間が推測 | **決定ジャーナル**（commit point の有無で決定的） |
| 追加コスト | なし | tx_journal への 1 write（happy 2pc に fsync 1 回増）+ 起動時スキャン + **回復ロジックの正しさ責任** |

## わかったこと
1. **in-doubt は「解けない問題」ではなく「決定を永続化し損ねている問題」**。2PC が宙吊るのは
   「coordinator の意思(commit or abort)が揮発メモリにしか無い」から。**commit point で意思をディスクに落とせば**、
   クラッシュしても起動時に「意思の通りに前進(roll-forward)」or「意思が無いから巻き戻し(roll-back)」で決定的に片づく。
   これはまさに **XA トランザクションマネージャ / DB 内蔵 2PC coordinator が内部でやっていること**を手で再現したもの。
2. **presumed-abort**: commit 決定だけ書き、abort は「行が無い＝abort」で表す。abort が多数派(prepare 途中失敗は abort)なので
   書込を節約できる古典的最適化。ただし裏返しに「**journal を消すタイミング**」が correctness の綱渡りになる(deferred-delete)。
3. **回復は "serve より先" に走らせる**。fire-and-forget にすると (a) tx_journal 作成(initDb)とレースして初回起動で
   サイレント no-op、(b) 多重 coordinator では live traffic の prepare 途中 tx を「決定なし→abort」と誤って巻き込む。
   **リゾルバは schema 初期化後・listen 前に await 完走**が正解(qa 指摘で fire-and-forget→await に修正)。
4. **タダではない**: 2PC は「動く」から「運用できる」に上がったが、その代償は **coordinator がステートフルになる**こと
   (自前の回復ログ tx_journal を持ち、その削除順序・未達時の deferred・serve 前実行という**非自明な不変条件**を守り続ける責任)。
   教訓38「タダの分散tx は無い」の続き —— **2PC を実運用に耐えさせる税は "ミニ・トランザクションマネージャの自作と保守"**。
