# Lv20 — 実ネットワークの税：分散tx を k3d に載せ替える —— 往復差は「平均」でなく「裾」に出る

記録日: 2026-07-15 / マシン: 8 コア Mac / **k3d(k3s)実クラスタ・2 agent ノード** / 計測時 load avg **11〜22（荒い）**

## やったこと
- Lv19 の 2DB 跨ぎ order(none/2PC/saga)を **docker-compose(同一ホスト bridge)から k3d 実クラスタへそのまま載せ替え**、
  同じ 1VU 逐次で再測。**app コードは一切変えていない**（純インフラの移植 = 測定台の入れ替え）。
- coordinator(`api`, SERVICE=all)と `items-service`(SERVICE=items)を **podAntiAffinity で別ノードに配置**。
  在庫調停の HTTP がノード境界を越え、k8s dataplane(flannel VXLAN + kube-proxy iptables + ClusterIP DNS)を通る。
  - 実配置: `api`@agent-0 / `items-service`@agent-1（`kubectl get pods -o wide` で NODE 差を確認）。
- 目的: 「Lv19 の docker bridge では往復が**ほぼタダ**だった。実ネットワークに載せると 2PC の余分な往復の税が見えるはず」を検証。
- **正直な枠付け**: 同一 Mac 上なので物理層は loopback。これは **k8s dataplane 税**であって WAN 遅延ではない。
  地理的 RTT を再現したければ `tc netem`(stretch・README 参照)。

# 数値は host load が 11→22 で揺れる中の median（テールは load に強く汚染される）
                    k3d median      （参考）Lv19 compose median   往復(items へのHTTP)
none :  17.9〜23.5ms                3.42ms                        1（decrement）
saga :  27.0ms                     5.04ms                        1（reserve）+ ローカルDB op 多め

compose 3〜5ms  →  k3d 18〜27ms   ≒ 5x       # bridge では見えなかった床が全モードに乗る

### 2pc のテール（load スパイク下で critical section が長い分だけ裾が爆発）###

| 指標 | none | saga | 2PC |
|---|---|---|---|
| compose median (Lv19) | 3.42ms | 5.04ms | 5.46ms |
| k3d median（静穏 load~12-18） | ~18-24ms | ~27ms | ~27ms |
| **compose 比の 2PC/none** | — | — | **1.6x** |
| **k3d 比の 2PC/none（静穏 matched-load）** | — | — | **1.16x** |
| k3d テール p95（load 競合時） | ~90ms | ~90ms | **1.38s** |
| in-doubt（fault 注入時） | リーク（ロック無） | 補償で回復（ロック無） | **prepared 宙吊り・行ロック** |

## わかったこと
1. **dataplane floor は全モード一律・往復本数に非依存**。compose 3-5ms → k3d 18-27ms(~5x)。
   k8s のネットワークは「ClusterIP を引く DNS + iptables 転送 + overlay ホップ」で、docker bridge の直結より 1 往復が重い。
   **この床が median を支配するので、往復本数の差は median の中では相対的に縮む**
   （2PC/none 比は compose 1.6x → k3d 1.16x）。「実ネットワークにすると 2PC が平均で急に遅くなる」は**起きない**——floor が皆に乗るから。
2. **2PC の税は median でなくテールと競合耐性に出る**。2PC は prepared txn を「余分な cross-node 往復」越しに保持する分
   critical section が長く、host が混むと p95 が 57ms→1.38s、max 3.73s まで跳ねる。none/saga は 1 往復で短く、裾が有界。
   → **「分散tx の実ネットワーク税は平均でなく裾に出る」**。SLO を p50 で切ると 2PC の危険が見えない。
3. **Lv19 の失敗モードは k3d 上でも同一に再現**。in-doubt は「両DBに prepared・行ロック・手動 ROLLBACK PREPARED」のまま。
   基盤を変えても**プロトコルの弱点は移動しない**（= Lv21 で coordinator 側に決定ジャーナル+起動時 resolver を入れて自動回復させる）。
4. **測定規律の再確認**: この段は host load が 11→22 で乱高下し(Xcode/Sim/Chrome)、median ですら ±40% 揺れた。
   **頑健だったのは「floor の存在(~5x)」「2PC テール爆発」「in-doubt 再現」の質的信号**で、
   none/saga/2PC の median 差(~1.5x 以内)は noise 隣接。数値の絶対値でなく、**モード間の質的差**を読むべき段。

## 一言でいうと
> **docker bridge が隠していた「1 往復の値段」を k8s dataplane が可視化する。**
> その床は全モードに一律に乗るので平均は横並びに近づき、**2PC の余分な往復は平均でなく「裾」と「競合耐性」を壊す**。
> 基盤を実ネットワークにしても in-doubt の宙吊りは消えない——プロトコルの弱点は引っ越しても付いてくる（→ Lv21 で潰す）。
