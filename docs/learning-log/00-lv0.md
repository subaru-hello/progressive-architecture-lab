# Lv0 — docker-compose（個人開発レベル）

記録日: 2026-07-06

## やったこと
- 共有アプリ `app/`（Fastify + node-postgres + prom-client, TypeScript）を作成
- `docker-compose.yml` で `api`（1 コンテナ）+ `db`（postgres:16）を起動
- ヘルスチェック / DB CRUD / Prometheus メトリクス / k6 負荷試験までを疎通

- k3s / k3d / k6 は未導入 → k3s は Mac で直接動かないため **Lv2 以降は k3d(=k3s in Docker)** を使う方針に。
- k6 も **`grafana/k6` の Docker イメージ**で実行し、ローカルインストールを避ける（実費ゼロ・環境を汚さない）。

## ビルド
```console
$ cd app && npm install && npm run build && ls dist
> pal-api@0.1.0 build
> tsc
server.js
server.js.map
```
- TypeScript は ESM（`"type":"module"` + `moduleResolution: NodeNext`）。相対 import の拡張子問題を避けるため、初手はサーバーを 1 ファイル `src/server.ts` にまとめた。

## 起動
```console
$ docker compose -f stages/00-local-compose/docker-compose.yml up --build -d
 Container pal-lv0-db-1  Healthy
 Container pal-lv0-api-1  Started
```
- `depends_on: { db: { condition: service_healthy } }` により、Postgres の `pg_isready` が通ってから API が起動する。
- さらにアプリ側でも `initDb()` を最大 30 回リトライ。**「起動順序は保証されない」前提でアプリを書く**のが後段（k8s）でも効いてくる。

## 負荷試験（k6 スモーク）
```console
$ docker compose -f stages/00-local-compose/docker-compose.yml up -d
$ docker run --rm -i grafana/k6 run - < load/smoke.js
    ✓ 'p(95)<500' p(95)=29.76ms
    ✓ 'rate<0.01' rate=0.00%
    checks_succeeded...: 100.00% 2186 out of 2186
    http_req_duration..: avg=25.38ms  p(95)=29.76ms  max=126.73ms
    http_req_failed....: 0.00%  0 out of 2186
    http_reqs..........: 2186   62.386975/s
    vus_max............: 10
running (0m35.0s), 00/10 VUs, 2186 complete and 0 interrupted iterations
```
- 同時 10 VU・`/work?ms=20`（20ms 待機）で **62 req/s、p95≈30ms、エラー 0%**。
- `ms=20` の I/O 待ちが律速なので 1 コンテナでも余裕。**CPU を食わせる `cpu=` に変えると 1 プロセスの限界が見える**はず（次段で検証）。

## 気づき / ハマり
- Mac の Docker では `--network host` が効かないため、コンテナ（k6）→ ホスト（API）は `host.docker.internal` で叩く。`load/smoke.js` は `BASE_URL` で切替可能にした。
- `reply.elapsedTime`（Fastify v4）で各リクエストのレイテンシを取り、`onResponse` フックで Prometheus ヒストグラムに記録。メトリクス基盤を最初の段から持つと、後段の比較が楽。

## 次へ進む理由（Lv0 の限界）
- `api` が 1 コンテナ。CPU 律速の処理が増えると頭打ち。
- `docker compose up --scale api=3` で増やしても、**手前に振り分ける LB が無い**ので実運用にならない（ホストポートの取り合いになる）。
- → **Lv1**: リバースプロキシ（Caddy / Traefik）を前段に置き、複数レプリカへ負荷分散。`instance` がバラけることを確認し、CPU 律速時にスケールで捌けるかを k6 で計測する。
