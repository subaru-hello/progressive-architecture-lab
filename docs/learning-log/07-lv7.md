# Lv7 — worker_threads：CPU 律速の「回避」でなく「根治」

記録日: 2026-07-11 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine + node:20-alpine

## やったこと
- **Lv2 の宿題への回答。** Lv2 で「CPU バーンが Node の単一イベントループを塞ぐ → `/health`・`/ready`
  が応答できず probe タイムアウト → k8s が健全な Pod を Evict/再起動して雪崩れる」を発見し、
  対処は probe 寛容化（回避策）で止めていた。真の解として予告した **worker_threads offload** を実装。
- CPU バーン（`/work?cpu=N` の `sqrt` ループ）を `worker_threads` の**常駐プール**に逃がし、
  イベントループを空ける。これで CPU 飽和中も `/health` が即応することを compose だけで（k8s 無しで）実測。
- アプリ変更（`app/src/server.ts` + 新規 `pool.ts` / `cpu-worker.ts`、後方互換: `WORKER_POOL_SIZE`
  未設定なら現状の同期パスのまま＝Lv0-6 の全 compose 無改変。Redis の `REDIS_URL` 分岐と同型）:
  - `app/src/pool.ts`: ゼロ依存の自作 `WorkerPool`（piscina 等は入れない）。起動時に size 本を常駐、
    job は連番 id + `Map` で resolve/reject を対応付け、空き worker に即割当／無ければ内部キュー。
  - **背圧**: `queueDepth >= WORKER_QUEUE_MAX`（既定 size×100）なら `/work` は即 **503**。無限キューは却下。
  - `app/src/cpu-worker.ts`: `parentPort` で `{id,cpu}` を受け sqrt バーン → `{id,sum}` を返す。
  - 可観測性: prom Gauge `worker_pool_busy` / `worker_pool_queue_depth`。
  - 実験の性質上 worker が焼き殺されうるので、**worker の error/exit で in-flight job を reject し
    worker を作り直す自己回復**を入れた（これが無いと死んだ worker にキューを積み `/work` が永久ハング）。

## 実測（cpu-vs-health.js / PEAK_VUS=100・CPU=10、wall↔worker を背中合わせで A/B）
`/health` を 10 req/s で叩き続けながら `/work?cpu=10` を 100 VU までランプ。api は **replicas: 1**
（単一プロセスで「同期バーンが同居 /health を飢えさせる」を純粋に見せる。LB で逃がさない）。

```console
### WALL (sync burn, WORKER_POOL_SIZE 未設定) ###
health p(95)=1s (1s timeout 上限に張り付き)  health med=827ms  health 失敗率=46.38% (205/442)
throughput=83 iter/s   work 503=0 (同期は落とさず全部詰まる)   (load avg 8.9)
### WORKER (WORKER_POOL_SIZE=4) ###
health p(95)=7.71ms   health med=2.32ms   health 失敗率=0%
throughput=360 iter/s  work 503=0 (この負荷ではキュー未飽和)    (load avg 3.6)
```

| | wall（同期バーン） | worker（offload） | 差 |
|---|---:|---:|---:|
| /health p95 | **1s（timeout 上限）** | **7.71ms** | イベントループ解放 |
| /health 中央値 | 827ms | 2.32ms | −99.7% |
| /health 失敗率 | **46.38%** | **0%** | probe 雪崩の有無 |
| throughput（iter/s） | 83 | **360** | ≈4.3×（4 コア並列） |

## いちばんの気づき — Lv3（回避）と Lv7（根治）の対比が本命
- **wall の `/health` は 46% が 1s で死ぬ。** これが Lv2 の probe 雪崩の正体そのもの——
  単一イベントループが同期 CPU バーンに占有され、同居する probe が応答できない。
  k8s ならここで readiness 失敗 → Evict → 残った Pod に負荷集中 → さらに Evict の連鎖に入る。
- **worker は `/health` p95 7.71ms・失敗 0%。** CPU バーンは 4 スレッドに逃げ、イベントループは空。
- Lv3 の HPA は「飽和する前に Pod を足す」＝**回避策**（複数ノードでコアを増やして CPU 律速を先送り）。
  Lv7 は「**1 プロセス・1 replica のまま**、CPU をイベントループから引き剥がす」＝**根治**。
  ＝ **水平スケール（Pod 増）とスレッド並列（worker 増）は直交する手段。** 前者は容量を足し、
  後者はイベントループを守る。probe 雪崩の本当の原因はコア不足でなく**イベントループの占有**なので、
  後者でしか根治しない。ついでに 4 スレッドが 4 コアを並列使用して throughput も 4.3 倍になった。

## 背圧の設計判断（この段が見せたかったこと）
- **無限キューは却下した。** イベントループを空けるだけなら無限キューでもいいが、それだと `/work`
  レイテンシが青天井に伸び、滞留 job でメモリも伸びる＝**壁が別の場所に移動しただけ**（Lv5 の学びの再演）。
  上限キュー＋溢れたら即 503 にして「捌けない負荷は落とす＝背圧」を見せた。これが Lv2/3 の
  probe 雪崩（過負荷時に全部を道連れ）への正しいアンチテーゼ。503 を返せば上位がリトライを制御できる。
- **worker の死は必ず起きる前提で書く。** CPU を焼く実験は worker をクラッシュさせうる。
  死んだ worker に job を割り当て続ける／in-flight を宙吊りにすると `/work` が永久ハングする
  （qa レビューで検出）。error/exit で reject＋worker 再生成の自己回復が必須。

## 正直な注記（計測条件）
単一 Mac 上の相対 A/B として読むこと。計測時 load avg 3.6〜8.9 と変動し、**wall 側の方がむしろ
高負荷（8.9）で測れており wall に不利な条件**。それでも 46% vs 0% の差が出たので結論は堅い。
`PEAK_VUS` を上げる（例 200〜）と worker 側でも 503 背圧が出はじめる（キュー飽和）。
worker_pool_busy はバーン中ほぼ 4（全スレッド稼働）に張り付く。

## 次へ進む理由
- CPU 律速は根治できた。残る本丸は **write のスケール**（キャッシュも worker も write を助けない）。
  非同期ジョブ/キュー、write-through、細粒度キー＋single-flight（スタンピード対策）が次の宿題。

## 気づき（LEARNINGS.md に反映済）
1. **probe 雪崩の根本原因はコア不足でなくイベントループの占有。** だから HPA（Pod 増）は回避策、
   worker_threads（offload）が根治。単一 replica・単一プロセスのまま /health 失敗率 46%→0%。
2. **「イベントループを空ける」だけでは半分。上限キュー＋503 の背圧まで込みで初めて解。** 無限キューは
   壁を別の場所（レイテンシ・メモリ）に移すだけ。捌けない負荷は落とすのが過負荷時の正しい振る舞い。
3. **自作 worker プールは「worker の死」を一級市民として扱う。** error/exit で in-flight を reject＋
   worker 再生成しないと `/work` が永久ハングする。CPU を焼く実験ではこの経路を高確率で踏む。
