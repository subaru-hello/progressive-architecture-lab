# Lv22 — transactional outbox：exactly-once「配送」は無い。exactly-once「効果」＝ at-least-once ＋ 冪等消費者

記録日: 2026-07-15 / マシン: 8 コア Mac / Docker Compose(2DB) / 機能テスト(exactly-once 検証)

## やったこと
- `none` モード = decrement(items へ HTTP)→ order insert(orders-db) の**別々の2書込**＝ **dual-write**。障害で
  「stock 減・order 無し(lost)」or「order 有・decrement 無し(ghost)」——「ローカル DB commit と外部通知」の間に原子性が無い。
- これを **transactional outbox** で潰した(`ORDER_TX_MODE=outbox`・app コードのみ・外部契約不変):
  1. **原子的 emit**: order 作成 = **同一 orders-db tx** で `orders INSERT` + `outbox INSERT(msg_id)`。dual-write 窓が消える
     (order とその「decrement せよ」メッセージは一緒に commit されるか一緒に消えるか)。
  2. **at-least-once 配送**: 別 poller(2s)が `delivered_at IS NULL` の outbox を items の receiver へ HTTP 配送、成功で mark。
     失敗/クラッシュしても未配送のまま残り、次tick/再起動で再配送 —— **配送は 1 回以上を保証**。
  3. **冪等 receiver**: items 側 `applyDecrementIdempotent` = **同一 items-db tx** で
     `INSERT processed_messages(msg_id) ON CONFLICT DO NOTHING` → 新規なら `UPDATE items stock`、既処理なら何もしない。
     dedup と stock 更新が同一 tx なので「記録済だが未適用」も「適用済だが未記録」も起きない。
- **exactly-once 効果 = at-least-once 配送 × 冪等消費者**。配送は何度でも起こりうるが、消費者が dedup するので**適用は 1 回**。

## コマンドと出力
```console
### DEMO① happy（原子的 emit → poller が配送 → stock 1回だけ減）###
POST /orders {itemId:3, qty:5} → 201  order#1                          # 即時: order 作成・stock 未変化
  即時:   orders=1  stock=1000000  outbox_undelivered=1                # emit 済(同一tx)・配送はこれから
  ~poller: stock=999995  outbox_delivered=1  processed=1               # 配送・stock -5・dedup 記録

### DEMO② exactly-once（FAULT after-deliver-before-mark = 配送後・mark前にクラッシュ → 再配送ループ）###
POST /orders {itemId:3, qty:5} → 201  order#2
  fault 継続中(poller が3回再配送・warn×3): stock=999990  outbox_undelivered=1  processed=2
    # ★ 3回配送されても stock は 999995→999990 の 5 だけ(=1回)。再配送は全て dedup で no-op。
  再起動(fault 解除) → 再配送→dedup(duplicate)→mark:
    stock=999990(据置)  outbox_undelivered=0  outbox_delivered=2  processed=2  orders=2
    # ★ stuck message が収束。stock は動かない。最終: 2注文=stock -10 ちょうど。
```

## none(dual-write) との対比
| | none（dual-write） | outbox |
|---|---|---|
| 「commit + 通知」の原子性 | **無い**（別々の2書込） | **有る**（order+outbox を同一tx） |
| 障害時 | lost（stock 減・order 無）/ ghost（order 有・decrement 無） | 無し（emit 原子的・配送 at-least-once・適用 冪等） |
| 配送保証 | at-most-once(失敗で消える) | **at-least-once**（未配送は残り再送） |
| 二重適用 | — | **dedup(processed_messages)で 1 回**に収束 |
| 代償 | 速いが壊れる | 非同期（配送は事後）・同期 stock チェックを捨てる（在庫は事後権威＝負在庫あり得る）・poller と2枚のテーブル |

## わかったこと
1. **「exactly-once 配送」は原理的に無い。有るのは「exactly-once 効果」＝ at-least-once 配送 × 冪等消費者**。
   ネットワークは「送った/届いた」を確実に知れない(2 generals)ので配送は再送前提。**受け手が dedup キーで冪等**なら、
   何回届いても効果は 1 回。outbox はこの「冪等キー(msg_id)」を**送り手のローカル tx で採番・永続化**する仕掛け。
2. **outbox は「DB commit と外部 emit の原子性」を"分散tx"から"単一DB tx + dedup"へ格下げする**。
   dual-write の原子性問題(2つのシステムに跨る)を、outbox は「ローカル1テーブル追加」で単一DB tx に閉じ込め、
   残りを「再送 × 冪等」で吸収する。**2PC(Lv19-21)が原子性をブロッキング(prepared ロック・in-doubt・coordinator 回復)で
   買ったのに対し、outbox はロックも in-doubt も持たず poller と dedup だけで買う** —— 多くの現代システムが emit に
   2PC でなく outbox を選ぶ理由。教訓38「タダの分散tx は無い」の"別の払い方"。
3. **代償は"同期の確実さ"を"非同期の結果整合"に替えること**。order は stock 確定前に 201 を返す(emit だけ原子的)。
   同期 stock チェックを落とすので在庫は事後権威(負在庫はビジネス判断)。速さと引き換えでなく「壁の位置」を替える取引
   (教訓33 の再演)。
