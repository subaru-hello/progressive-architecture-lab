# Lv18 — ダウンタイムゼロ抽出（strangler-fig）：hex seam から items を live で移す

記録日: 2026-07-13 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~5-6

## やったこと
- Lv17 の hex seam（orders→items が ItemsPort interface 1 本）から、**items を専用サービス+専用DBへライブ抽出**。
  **Lv16 の泥団子で 100% ダウンした同じ引っ越しを、今度は k6 を常時流して error≈0 で通す**。
- **再起動ゼロで phase を切替**するため、`DualWriteItemsAdapter` に **runtime の migration mode**（admin endpoint で切替）を実装:
  `primary_only`(baseline) → `dual_write`(両書き・primary 権威) → `dual_shadow`(影読み比較) → `secondary_only`(cutover)。
  env でなく runtime 切替なので、各 phase 遷移でコンテナを作り直さない＝**真のゼロダウンタイム**。
- **strangler フル playbook**（全て monolith への admin curl・k6 は流しっぱなし）:
  1. baseline: orders→items は in-process primary。items-service は空。
  2. **dual_write ON（backfill の前に）**: 在庫デクリメントを primary + items-service の両方へ（primary 権威・secondary は
     best-effort mirror）。**backfill より先に有効化**して「snapshot 後の書き漏れ（drift 窓）」を塞ぐ。
  3. **backfill**: monolith `/internal/items/all` → items-service `/internal/items/bulk-upsert`（冪等・id/created_at 保存・
     SERIAL sequence を setval で補正）。dual_write が並行で収束させる。
  4. **dual_shadow**: 読みは primary 権威のまま items-service も影読みして比較 → `shadow_mismatch` を計上。**整合を確認する検証ゲート**。
  5. **cutover（secondary_only）**: 読み書きを items-service に切替（admin で mode flip・再起動なし）。items-service が権威に。
  6. contract: dual-write 撤去・monolith items 休眠（概念）。

## 実測データ（引っ越し中の失敗率＝ダウンタイム）
| アーム | items ライブ抽出中の http_req_failed | 備考 |
|---|---|---|
| Lv16 泥団子（naive 移動） | **100%**（843/843） | 跨ぎ FK/直 JOIN が新 DB を跨げず全滅 |
| **Lv18 hex 規律（フル playbook）** | **0%（全 phase）** | shadow_mismatch=0 で整合確認後に cutover |
| Lv18 hex naive（手順飛ばし） | **66.7%**（全 order 409） | seam はあっても playbook を抜くと壊れる |

## いちばんの気づき — 同じ引っ越し・違う下地。ゼロダウンタイムは「seam × 手順」の積
1. **Lv16 で 100% ダウンした同じ抽出が、hex seam では 0% で通った**。差は下地（結合度）——orders→items が
   ItemsPort 1 本だから、アダプタを in-process→dual-write→HTTP と runtime で差し替えるだけで、プロセスを止めずに
   移行できる。**hexagonal 境界が「事前に引いた切断線」だったからライブで切れた**（教訓36 の実地回収）。
2. **seam だけでは足りない・手順も要る**。同じ hex でも naive（dual_write/backfill/shadow を飛ばして即 cutover）は
   **66.7% ダウン**（空の items-service へ切替＝在庫があるのに全 order が 409）。ゼロダウンタイムは
   **「切れる構造(seam)」×「切る手順(expand→dual-write→backfill→verify→cutover→contract)」の積**。片方でも欠けると落ちる。
3. **verify(shadow-read)が cutover の安全装置**。`shadow_mismatch=0` を確認してから切替えたから 0% で通せた。
   「移す前に、新ストアが本当に一致しているかを影読みで確かめる」——推測でなく計測で cutover の可否を決める（教訓6 の移行版）。
4. **runtime mode 切替が「ゼロ再起動」の肝**。phase を env でやるとコンテナ再作成＝ダウンタイム。admin endpoint で
   アダプタの mode を差し替えると、**プロセスを止めずに** primary→dual→shadow→secondary を渡れる。

## 払ったコスト / トレードオフ
- **移行機構の実装**（dual-write アダプタ・backfill・shadow・runtime mode・admin）。移行のためだけの一時コード＝
  cutover 後に contract で撤去する前提。
- **dual-write は非冪等**: order リトライで両ストアが二重デクリメントし drift しうる（shadow が検出）。冪等化は Lv19 の宿題。
- **drift 窓**: backfill snapshot 後・dual_write 前に書くと漏れる → **dual_write を先に**有効化する順序が必須（runbook）。

## 実装メモ / 罠
- builder 実装 → qa-reviewer で「dual-write の authority/ordering・shadow 比較・backfill 冪等/sequence・created_at 保存・
  mode の torn read」を確認。must-fix: **bulkUpsert を tx で包む**（INSERT+setval の部分失敗で sequence がずれる）を是正。
- **cutover(secondary_only)は残った cross-DB 原子性を露出する**: 在庫は items-db、order は orders-db の別 tx → Lv15 教訓34 の
  「原子性が自前の分散tx問題に化ける」窓が、これで本番化する。→ Lv19 で none/2PC/saga を実測。
- 計測時 load 5-6（median を頑健信号に）。naive アームは破壊的でなく mode を戻せば回復（items-service を空で再現）。

## 次へ（Lv19）
- items が別サービス+別DBになった今、POST /orders の在庫デクリメント(items-db)+order INSERT(orders-db)は2DB跨ぎ。
  `ORDER_TX_MODE=none|2pc|saga` を障害注入で比較し、**タダの分散tx は無い**（2PC=原子性をブロッキング/in-doubt で買う・
  saga=可用性を補償/冪等の実装コストで買う）を実測する。
