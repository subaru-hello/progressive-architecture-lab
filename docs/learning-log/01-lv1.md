# Lv1 — リバースプロキシ + 複数レプリカ

記録日: 2026-07-06 / マシン: 8 コア

## やったこと
- 前段に **Caddy** を置き、`api` を複数レプリカ化。1 ホストポート(:8080)で受けて全レプリカへラウンドロビン。
- Lv0 の「`--scale` するとポート衝突で死ぬ」を解決。
- Lv0 と同じ CPU 負荷をかけ、レプリカ数でスループットがどう伸びるか実測。

## 構成
```
localhost:8080 --> caddy --round robin--> api x N --> db
```
Caddy の `dynamic a`（Docker DNS を定期再解決）で、レプリカ増減に追従して振り分け先を自動更新。


### ロードバランシングの目視
```console
$ for i in $(seq 1 12); do curl -s localhost:8080/health | jq -r .instance; done | sort | uniq -c
```

### 負荷試験（cpu=10, 20VU）
```console
# x3
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8080 -e CPU=10 grafana/k6 run - < load/cpu.js
    http_req_duration..: avg=71.54ms  p(95)=186.82ms  max=517.35ms
    http_reqs..........: 5561   276.23686/s

# x6（--scale api=6 で増やして再計測）
$ docker compose -f stages/01-compose-proxy/docker-compose.yml up -d --scale api=6
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8080 -e CPU=10 grafana/k6 run - < load/cpu.js
    http_req_duration..: avg=50.49ms  p(95)=214.39ms  max=1.54s
    http_reqs..........: 7849   390.051478/s
```

## 結果（Lv0・番外編との比較）
| 構成 | スループット | p95 | 備考 |
|---|---:|---:|---|
| Lv0 Node x1 | 98 req/s | 372ms | 1コア頭打ち |
| **Lv1 Node x3** | **276 req/s** | **187ms** | 約2.8倍・p95半減 |
| Lv1 Node x6 | 390 req/s | 214ms | max 1.54s（テール悪化） |
| (参考) Go 1proc@8core | 303 req/s | 106ms | 番外編 |

## 気づき
1. **言語を変えず、レプリカ x3 だけで Go@8（303 req/s）に並んだ**（276 req/s）。番外編の結論「Node の頭打ちは言語ではなく1プロセス=1コア。横に増やせば多コアを使える」が、そのまま数値で実証された。
2. **x6 で 390 req/s まで伸びたが、p95/max が悪化**（max 1.54s）。8 コアに対しプロセスを積みすぎると **オーバーサブスクリプション**でテールが荒れる。「レプリカは多いほど良い」ではない。
3. Caddy の `dynamic a` は Docker DNS を引き直すので、`--scale` の増減にプロキシ設定を触らず追従できた。

## 次へ進む理由（Lv1 の限界）
- レプリカ数・復旧・スケールが**手動 / 命令的**。落ちた時の自動復旧も、負荷に応じた自動増減もない。
- スケールは **1 台のコア数が上限**。これ以上は **マシンを増やす**しかない。
- → **Lv2**: 同じアプリを k3s(k3d) に載せ、Deployment/Service/Ingress で**宣言的**に運用。**Lv3** で複数ノード + HPA の自動スケールへ。
