# Lv6 — Redis キャッシュ：read 負荷は「移す」でなく「消す」

記録日: 2026-07-10 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine + redis:7-alpine

## やったこと
- Lv5 の宿題への回答。Lv5 は「read replica は単一ホストだとコア共有で効かない（負荷が別コンテナに移動するだけ）」で終わった。
- 同じ read-heavy 負荷に **Redis キャッシュ（cache-aside）** を足し、read 負荷を**移す**のでなく**消す**とどうなるかを実測。
- アプリ変更（`app/src/server.ts`、後方互換: `REDIS_URL` 未設定なら Redis に触れず Lv0-5 は従来通り）:
  - `GET /items`: Redis GET → hit ならそのまま返す（DB を叩かない）/ miss なら readPool の SELECT → `SET items:latest100 <json> EX 30` → 返す。`instance` はキャッシュに混ぜず毎回現在値で包む（LB 可視化維持）。
  - `POST /items`: INSERT 後に `DEL items:latest100`（write-through でなく invalidation）。
  - Redis 障害耐性: `lazyConnect / maxRetriesPerRequest:1 / enableOfflineQueue:false / connectTimeout:2000` で「速く失敗して DB フォールバック」。全キャッシュ操作は try/catch。
  - 可観測性: prom counter `cache_hits_total`/`cache_misses_total` と eyeball `GET /cache`。

## 実測（read95:write5、wall↔cache を背中合わせで A/B）
```console
### WALL (no cache) ###
wall150: read p(95)=161ms  throughput=1037/s  failed=0.00%  (load avg 6.2)
wall400: read p(95)=636ms  throughput=1207/s  failed=0.00%  (load avg 9.8)
### CACHE ###
cache150: read p(95)=98ms  throughput=1581/s  failed=0.00%  (load avg 10.5)
cache400: read p(95)=99ms  throughput=3499/s  failed=0.00%  (load avg 13.8)
# 全体 hit 率（Redis keyspace）: @150 52601/(52601+15132)=77.6% / @400 113906/(113906+35779)=76.1%
```

| | wall（cache 無） | cache | 差 |
|---|---:|---:|---:|
| 150 VU read p95 | 161ms | **98ms** | −39% |
| 150 VU throughput | 1037 req/s | **1581 req/s** | +53% |
| 400 VU read p95 | **636ms** | **99ms** | **−84%** |
| 400 VU throughput | 1207 req/s | **3499 req/s** | **+190%** |
| cache hit 率 | — | **~76%** | — |

## いちばんの気づき — Lv5 との対比が本命
- **400 VU で wall は DB が飽和し read p95 が 636ms まで崩れる。** cache は hit がその読み取りを
  肩代わりするので **read p95 は 99ms で平坦、throughput は約 3 倍**。
- Lv5 の read replica は @400 で read p95 を改善しなかった（147ms、負荷を別コンテナに**移動**しただけ、
  同じコアを共有するので箱全体は CPU 律速のまま）。**キャッシュはヒット時に DB クエリを消す**
  （Redis の in-memory GET は Postgres の sort/scan より桁違いに軽い）ので、共有コア上でも**仕事が減る**。
  ＝ **「負荷を移す」施策（replica）と「負荷を消す」施策（cache）は、単一ホストで効果が正反対に出る。**

## 正直な注記（計測条件）
計測中マシンは他プロセス（ブラウザ）で **load avg 6〜14 と変動**。しかも cache 側の方が高負荷時に
測れており（cache400 は load 13.8）**キャッシュに不利な条件**。それでもこの差が出たので結論は堅いが、
**絶対値でなく wall↔cache の相対 A/B** として読むこと（教訓6「throughput 単体は嘘をつく」の運用）。
最初 cache と wall を別時刻で測ったら wall が異常に遅く「勝ちすぎ」に見えた → `uptime` の load avg で
Chrome が犯人と判明 → **背中合わせで測り直した**。計測条件を揃えないと差が捏造になる好例。

## キャッシュの難所（この段が見せたかったこと）
1. **hit 率は read 比率でなく「無効化頻度」で決まる。** read95/write5 でも write ごとに単一キーを
   全消しするので、高負荷では ~7ms に 1 回キャッシュが飛ぶ。だから hit 率は 0.95 でなく **~76%**。
   DEL 直後は全レプリカが同時 miss して DB に殺到する（**スタンピード**）。
2. **read-after-write は DEL では保証されない（cache-aside の競合）。** 並行 GET が INSERT 前の DB を
   読み、その古いスナップショットを DEL の**後**に SET すると、TTL 切れまで古い値が残る。
   逐次なら反映されるが並行下では保証なし。「DEL すれば即整合」は嘘。TTL 30s は最終収束の安全網。
3. **Redis はベストエフォート層に留める。** `/ready` に含めず、接続エラーは DB フォールバック。
   キャッシュは可用性の依存先でなく最適化。Redis を止めても 200 で応答（検証済み）。

## 次へ進む理由
- read はキャッシュで消せた。残る本丸は **write のスケール**（キャッシュは write を助けない）と、
  Lv2/5 で予告した **CPU 律速の真の解（worker_threads / オフロード）**。
- キャッシュ整合性を突き詰めるなら write-through / 単一キー→細粒度キー / single-flight（スタンピード対策）が次の宿題。

## 気づき（LEARNINGS.md に反映済）
1. **「負荷を移す」と「負荷を消す」は単一ホストで効果が正反対。** replica(Lv5)は read p95 不変、cache(Lv6)は −84%・throughput +190%。ヒットは DB クエリ自体を消すのでコア共有の罠を受けない。
2. **キャッシュの効果は read 比率でなく無効化頻度で決まる。** write5% でも DEL 全消しで hit 率 ~76% 止まり、DEL 直後はスタンピードで DB に殺到。
3. **cache-aside の DEL は read-after-write を保証しない。** 並行 GET の stale SET が DEL 後に残る競合。TTL が最終収束の保険。キャッシュ整合の一番の難所。
