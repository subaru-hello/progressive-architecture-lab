# Lv13 — monolith（多ドメイン app のベースライン）：cross-context を「直 SQL JOIN」で in-process 解決する

記録日: 2026-07-12 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~3-5

## やったこと
- 分解軸（monolith ↔ modular ↔ micro）を「本物」に測るための**前提投資**として、単一 `items` ドメインだった
  app を **items + orders + users の 3 ドメイン**に拡張。**cross-context 依存 2 本**を作る:
  - **orders → users**（認証）: `POST /orders` は `x-auth-token` を users で検証してから受ける。
  - **orders → items**（在庫）: order 作成時に item の在庫を参照・デクリメントする。
- **ポート&アダプタで「app 全段不変」を守る**（`app/src/pool.ts` の `WorkerPool` 抽象と同じ精神）。
  ドメインロジックは一度だけ書き、cross-context を **ポート interface** 越しにして、**アダプタ差し替えだけで
  monolith/modular/micro を実現**（ドメインコードは 3 形態で 1 行も変えない）。基盤だけ env で切替える既存規約を踏襲。
- **Lv13 = monolith**: 1 プロセスに 3 ドメイン同居・共有 1 Postgres。`ORDERS_CROSS_CONTEXT=join` で
  orders が **items テーブルを直接 SQL で JOIN/UPDATE** して在庫を引く（＝schema 結合・全ドメインが 1 DB を共有）。
  in-process なので order 作成 latency は最小。**ここを基準に Lv14/15 の対価を測る**。
- **後方互換**: `GET /items`(list) は `{id,name,created_at}` のみ返し stock を出さない → Lv0-12 の compose/image は不変。
  stock は新設 `GET /items/:id` に隔離。seed は `SEED_DEMO=1` の段だけ走る（旧段の空リスト baseline を守る）。

## 環境 / セットアップ
```bash
docker compose -f stages/13-monolith/docker-compose.yml up --build -d
# env: SERVICE=all, ORDERS_CROSS_CONTEXT=join, SEED_DEMO=1（demo item×50 在庫1e6 / demo user×2 を seed）
```

## いちばんの気づき — monolith の強みは「cross-context が関数呼び出し / DB が 1 つ」
1. **order 作成が最速・最単純**。auth も在庫参照も同一プロセス・同一 DB 内なので、`createOrder` は
   「1 トランザクションで items を JOIN/UPDATE して orders に INSERT」する 1 往復で済む。med 5.34ms。
   **これは分解の対価を測るための「税ゼロ」基準点**。
2. **原子性がタダで手に入る**。在庫デクリメントと order INSERT が同じ DB・同じ tx なので、
   在庫不足は `stock>=qty` ガードで丸ごと ROLLBACK（負在庫も宙吊り order も出ない）。
   Lv15 で DB を割ると、この原子性が最初に失われる（→ 15-lv15.md 教訓）。
3. **代償は結合**。orders が items テーブルの schema を直接知っている（直 JOIN）。速さと引き換えに、
   items の内部 schema を変えると orders が壊れる。**この結合を runtime コストゼロで切るのが Lv14**。

## 払ったコスト / トレードオフ
- **schema 結合**: orders ↔ items が DB テーブル越しに密結合。独立デプロイ・独立スケール不可（1 プロセス）。
- **障害の粒度が粗い**: プロセスが落ちれば 3 ドメイン全滅（blast radius = 全体。→ Lv15 で隔離を測る）。

## 実装メモ / 罠
- **後方互換の要**: `GET /items` list に stock を混ぜない・seed を `SEED_DEMO` でゲート。これで旧 Lv0-12 の
  再現性（空リスト baseline）を保つ。既存 pgdata に stock 列は `CREATE TABLE IF NOT EXISTS` で足されないが、
  旧段は orders を積まず stock を触らないので無害（新段は fresh volume）。
- **seed race**: 共有 DB に複数レプリカが同時 seed しうる段（Lv3/6 等）向けに、items seed は
  `pg_advisory_xact_lock` で直列化（レビュー指摘）。
- **qc**: builder 実装 → qa-reviewer 独立レビューで seed race / join+HTTP 誤設定 / 整数検証を是正してから計測。

## 次へ（Lv14）
- monolith の結合（orders が items テーブルを直参照）を、**同じ 1 プロセス・同じ DB のまま**、
  in-process の **ItemsPort（関数呼び出し）越し**に付け替える（`ORDERS_CROSS_CONTEXT=port`）。
  topology は不変・env 1 個差。**latency は変わらないはず**——変わるのは構造（結合の切断）。
  「runtime 無料で schema 結合を切れるか」を測る。
