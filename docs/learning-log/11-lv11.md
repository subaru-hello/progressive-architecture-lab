# Lv11 — 並列 committer + group commit（commit_delay）：durability を保ったまま fsync を束ねる

記録日: 2026-07-12 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine + redis:7-alpine + Node consumer

> ⚠️ **この段は実装・レビュー完了だが A/B 実測は未実施（延期）。** 記録日時点でマシンが Spotlight の
> フルインデックス再構築（`mdworker_shared`）＋ 8 ユーザーの GUI 負荷で load avg 30–48 を行き来し、
> SIP により Spotlight 停止も不可だった。ambient load が高い状態で測ると、単一 consumer の commit 挙動が
> CPU スケジューリング待ちに埋もれ、**group commit の効果が測れず両アーム（Lv9/Lv11）とも CPU 争奪で頭打ち**に
> なる（＝測っても学びが出ない）。よって「load<5 の静かな窓」が取れるまで実測を延期した。下記は設計・仮説と
> 実測手順。数値が埋まっていない表は意図的な空欄。

## やったこと（実装）
- **Lv9 = 律速は commit の「回数」、Lv10 = fsync の「コスト」は synchronous_commit=off で削れるが durability を失う。
  かつ「単一 committer では commit_delay が発火しない」**——この宿題を回収する段。
- consumer は Lv9 の multi-row INSERT のまま、**durability を保ったまま（synchronous_commit=on のまま）fsync を
  amortize できるか？** を問う。手段は PostgreSQL の group commit（`commit_delay` / `commit_siblings`）。
- アプリ変更（後方互換: 未設定なら Lv9/Lv10 とバイト同一）:
  - `app/src/consumer.ts`:
    - `COMMITTERS=N`（既定 1）: `main()` で `consumeLoop` coroutine を N 本 `Promise.all` で回す。Lv9/10 の
      consumeLoop は `processBatch` を await してから次を読む＝**commit は常に 1 本ずつ = 単一 committer**。
      commit_delay は「commit record を書いた瞬間に他に commit_siblings 本のアクティブ tx がある」ときだけ
      短い待ちを入れて WAL flush を 1 fsync に束ねる仕組みなので、単一 committer では原理的に発火しない。
      N 本が別々の pg コネクションで同時に commit へ到達して初めて group commit の母集団ができる。
    - `COMMIT_DELAY_US`（既定 0）/ `COMMIT_SIBLINGS`（既定 5）: `writePool` の libpq startup option
      `-c commit_delay=.. -c commit_siblings=..` として渡す（Lv10 の `synchronous_commit=off` と同じ options 合成に統合）。
    - **commit_delay は synchronous_commit=on のときだけ意味を持つ**（off だと backend は fsync を待たないので
      束ねる対象が無い）。だから Lv11 は `SYNC_COMMIT_OFF` を付けない＝Lv10 と違い durability を落とさない。
- **QA 指摘で本体修正した設計上の肝**:
  - 当初は N 本の committer が**単一 ioredis コネクションを共有**していた。ioredis は 1 インスタンス = 1 コネクション
    = 厳密 FIFO のコマンドキューなので、1 本の `BLOCK` 付き XREADGROUP がキュー先頭に居座ると、他 committer の
    read も INSERT 後の xack すら送信できず**直列化して「同時 commit」が崩れる**（＝Lv11 の前提が成立しない）。
    → 各 committer に**専用 read コネクション（`redis.duplicate()`）**を割り当てて解消。xdel/xack/del など
    非ブロッキング系は共有 `redis` のまま（BLOCK しないので相互ブロックしない）。COMMITTERS=1 のときは専用
    コネクションを張らず共有 `redis` を使い、Lv9/10 とコネクション構成もバイト同一に保つ。
  - `COMMITTERS` の NaN ガード: 非数値を渡すと `Array.from({length: NaN})` が空配列 → committer が 1 本も起動せず
    drain がサイレント停止する。COMMITTERS だけは明示的に 1 へフォールバック。

## 仮説（実測で検証すること）
1. **group commit が効くには並列 committer が要る（Lv10 の裏返し）。** ① 単一 committer では commit_delay が
   発火せず、② N committer で初めて複数 commit が WAL flush 点に同時到達して fsync が束なる。② が ① より
   drain 増・commit lag 減なら仮説成立。
2. **勝因の分離。** ②（committer 並列 + commit_delay）と ③（committer 並列のみ）の差分が group commit の純効果。
   ②≈③ なら「効いたのは並列化であって commit_delay ではない」= Lv8 の consumer scale-out の焼き直し。
   ②>③ なら commit_delay が上乗せしている＝group commit の実在確認。
3. **Docker-on-Mac の減衰。** fsync が VM 仮想ディスク経由で本番 SSD より鈍いため、group commit の効果も
   ベアメタルよりズレて（おそらく控えめに）出る見込み（Lv10 と同じ注記）。
4. **単一ホストの天井。** そもそも 800VU で箱全体が CPU 飽和する環境では、committer を増やしても CPU コアの
   取り合いになり、fsync 節約分が CPU 争奪に食われて効果が出にくい可能性がある（Lv3/Lv5 の「コア共有＝
   負荷を移すだけ」の罠の再演）。**本当に効かせるには fsync が律速で CPU に余裕がある状態が要る。**

## トレードオフ（設計上わかっていること）
- **durability は保つ**（Lv10 と違い synchronous_commit=on のまま）。commit は fsync 完了を待って返るので、
  クラッシュで commit 済みが消える窓は広げない。「durability を落とさず fsync を amortize する」のが Lv11 の売り。
- **代償は commit_delay 分のレイテンシ上乗せ**（siblings 未達なら待たないが、達したとき最大 commit_delay us）。
  低負荷では commit_delay が無駄待ちになりうる（commit_siblings で「他に十分な同時 tx があるとき」に限定して緩和）。
- 並列化による poison batch / PEL の増幅は「バッチ単位」のまま（COMMITTERS 倍にはならない）が、DB 全落ち時は
  同時に最大 COMMITTERS 本のバッチが PEL に積まれるので回収量は増える。at-least-once 不変量・XDEL→XACK 順は不変。

## 次の壁（Lv12 への引き）
- ① 実測で group commit の純効果を確定させるのが最優先の宿題。② その先、単一ホスト・単一ディスクでは fsync 帯域
  そのものが共有資源なので、**write スループットを本当に増やすには別ハード（別ディスク/別ノード）が要る**という
  Lv3/Lv5 の結論に再び突き当たる見込み。read replica / 物理シャード / 別ノードへの write 分割が候補。
