# Lv10 — synchronous_commit=off：commit の「コスト」を削る（が、回数削減には敵わない）

記録日: 2026-07-11 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine + redis:7-alpine + Node consumer

## やったこと
- **Lv9 の結論 = 律速は commit の「回数」（1 commit ごとの fsync/WAL flush 固定費）。** バッチ化はこの回数を
  N→1 に減らした。Lv10 は残る「1 回あたりの fsync コスト」を正面から攻める。
- consumer は Lv9 の multi-row INSERT のまま、**write セッションだけ `synchronous_commit=off`**。
  commit が WAL の fsync 完了を待たずに返る＝commit の cost を削る（回数は変えない）。
- アプリ変更（後方互換: `SYNC_COMMIT_OFF` 未設定なら Lv9 と同一）:
  - `app/src/consumer.ts`: `SYNC_COMMIT_OFF=1` のとき `writePool` に libpq startup option `-c synchronous_commit=off`
    を渡す。接続確立時点でサーバ側セッションに効くので `pool.on('connect')` で SET する方式のような
    「SET 完了前に最初のクエリが走る」レースが無い。**write 経路（consumer）だけ**に効き read 経路には波及しない。
  - 単一 consumer = 単一 committer なので `commit_delay`（group commit）は触らない——**並列 committer がいて初めて
    fsync をまとめられる**ので、単一 consumer では効かない（これも学び）。

## 実測（背中合わせ A/B、consumer 1 固定）
fsync の効果は「バッチ化で amortize 済みか」で激変するので **2 領域**で測った。
```console
### ① batched（Lv9 のまま）@800VU ― fsync は既に ~59:1 に amortize 済み ###
sync_commit=on : peak depth 1,203  drain 223,294 行/3,783 commit(avg 59)  commit lag ~110ms  503 0%
sync_commit=off: peak depth   362  drain 261,003 行/11,664 commit(avg 22) commit lag ~27ms   503 0%
  → depth 3.3×低・lag 4×低・drain +17%。効果は modest（バッチ化が既に fsync を薄めた後だから）

### ② per-row（BATCH_WRITES off）@400VU ― fsync が amortize されない領域 ###
sync_commit=on : peak depth 100,067(上限張付) drain 31,143 行  commit lag ~59s  503 43.17%
sync_commit=off: peak depth 100,085(上限張付) drain 47,241 行  commit lag ~43s  503 38.04%
  → drain +52%（fsync は per-row commit コストの相当部分だった）。だが depth は上限のまま・38%なお503
```
> ※ @800VU は箱全体が CPU 飽和（load avg 30–48）し producer も律速される。数値は各 A/B 内の相対差で読む。
> Docker-on-Mac は fsync が VM 仮想ディスク経由で本番 SSD より効きが鈍く、off の効果は控えめに出ている可能性大。

## いちばんの気づき — 同じ fsync 壁でも「回数削減」が「コスト削減」を圧倒する
1. **fsync は実在するコスト。** 外すと per-row で drain +52%、batched で +17%。commit lag も per-row 59→43s、
   batched 110→27ms と素直に下がった。「synchronous_commit=off は速い」は本当。
2. **だがバッチ化（Lv9）の方が桁違いに強いレバー。** バッチ化は 400VU で drain 6.4×（Lv9）。sync-off は
   せいぜい 1.5×（per-row）〜1.17×（batched）。**同じ fsync という壁を、回数を減らす方が cost を減らすより
   遥かに効率よく攻める**——1 fsync を 59 行で割る方が、1 fsync を「待たない」より効く。
3. **sync-off 単独では regime を変えられない。** per-row は sync-off にしても depth が上限 100k に張り付き 38% が
   なお 503。fsync を外しても per-row の他の固定費（parse/plan・トランザクション begin/end・プロトコル往復）は
   残り、単一 per-row consumer は producer を抜けない。**飽和から抜け出させたのは Lv9 のバッチ化であって sync-off ではない。**
4. **二つは同じボトルネックへの substitute（代替）レバー。** バッチ化＝fsync の「回数」を削る、sync-off＝fsync の
   「cost」を削る。既にバッチ化で回数を 59:1 に畳んだ後は sync-off の伸びしろは小さい（②→①で効果が薄れる）。
   **どのレバーが効くかは「ボトルネックがまだそこに残っているか」で決まる**（Lv6/Lv8 の教訓26 の再演）。

## 払ったコスト / トレードオフ
- **durability の緩和。** `synchronous_commit=off` は commit を返した後で WAL を fsync する。OS/DB クラッシュで
  直近数百 ms 分（`wal_writer_delay` 既定 200ms 程度）の **commit 済みトランザクションが失われうる**。
  データ破損ではなく最新分の消失。Lv8 の 202（受理≠永続化）に続く「速さと引き換えに何を失うか」の系譜。
- この consumer は「INSERT 成功で返った」時点で XDEL/XACK する。sync-off は「Stream から消したのに DB 側が
  crash で消える窓」を広げるが、**新たな整合性バグは生まない**（Lv8 で受容済みの at-least-once トレードオフの延長）。
  XDEL→XACK 順（depth 不変量の自己修復）は不変。

## 実装メモ
- pg（pure-JS client）は config の `options` を StartupMessage の `options` に載せてサーバへ渡す
  （`node_modules/pg/lib/client.js`）。DATABASE_URL 側に synchronous_commit を書くより、pool option で
  write 経路だけに限定する方がクリーン。`...(cond ? {options} : {})` で未設定時は Lv9 とバイト同一。

## 次の壁（Lv11 への引き）
- fsync コストも回数も削り切った先で単一 DB の write 帯域を更に上げるには、もう「同じ DB を効率化する」
  レバーは尽きつつある。次は **物理的に write を分割する**（partition/shard）か、**consumer を並列化して
  複数コネクションで同時 commit → group commit を効かせる**（ここで初めて commit_delay が意味を持つ）方向。
  ただし単一ホストでは Lv3/Lv5 の「コア共有＝負荷を移すだけ」の罠が待つ——本当に write 帯域を増やすには
  別ハードが要る、を再び踏むはず。
