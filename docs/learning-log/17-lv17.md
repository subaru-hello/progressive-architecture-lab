# Lv17 — 解きほぐし（mud → hexagonal: DB/Domain/Usecase/Infra）：結合を seam 1 本に落とす

記録日: 2026-07-13 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~5-6

## やったこと
- Lv16 の泥団子を **hexagonal(ports & adapters)** に解きほぐした（`ARCH=hex`＝既定）。ドメインごとに層を明示:
```
domains/<d>/  domain/(純 entity+不変条件・I/O ゼロ)  ports.ts(RepoPort interface)
              infra/pg-*-repo.ts(RepoPort を pg で実装)  usecase/(オーケストレーション)  routes.ts(HTTP アダプタ)
```
- **結合を落とす具体手**（Lv16 の絡みを 1 本ずつ切る）:
  1. **cross-domain JOIN(GET /orders) → 廃止**。orders は raw な orders 行だけ返す（enrich しない）。
  2. **跨ぎ FK(orders→items/users) → 廃止**。各ドメインが自分の表だけ所有。参照整合はアプリ/結果整合へ。
  3. **3テーブル共有 tx → usecase のオーケストレーション**。orders usecase が **ItemsPort/UsersPort interface** 越しに
     在庫・認証を得る（port モード）。join モードの `createWithDirectStock` は同一 DB 最適化として `OrdersRepoPort` 裏に隔離。
  4. **logic → usecase / SQL → infra / domain は純化**。`routes.ts` は「parse→usecase→エラーを status に写す」だけ。
- **依存の向き**: `domain/*` は I/O を一切 import しない（pg/fetch/fastify/redis ゼロ）。usecase は **port interface** に依存し、
  infra 具象は composition root(server.ts)だけが知る。
- **正直な設計判断2つ**:
  - **thin CRUD には usecase 層を作らない**。items/users は routes→repo(port) 直（オーケストレーションが無いので passthrough usecase は
    ノイズ）。**usecase 層はオーケストレーションがある orders でこそ価値を持つ**（cargo-cult しない）。
  - **「在庫充足」を純ドメイン述語(canFulfil)にしない**。check-then-act の race になるため。並行性を持つ不変条件は
    infra の atomic guard(`UPDATE … WHERE stock >= qty`)が担う。純 domain に残すのは `assertPositiveQuantity` のような
    並行性に依らない不変条件だけ。

## 実測データ（結合度 before/after）
| 指標 | Lv16 mud | Lv17 hex |
|---|---|---|
| cross-domain 共有 tx（3テーブル横断） | **1** | **0**（orders は orders を書く。join の UPDATE items は port 裏の最適化） |
| cross-domain JOIN | **1** | **0** |
| 跨ぎ FK | **2** | **0** |
| cross-context の表現 | あちこちに絡む | **ItemsPort/UsersPort という 1 本の seam** |
| order_latency med | 5.85ms | **4.85ms**（＝差なし・層化は runtime 無料） |

## いちばんの気づき — 綺麗にする＝「切る場所に seam を仕込む」こと。hexagonal 境界＝事前の切断線
1. **結合が seam 1 本に激減**。mud の 共有 tx/JOIN/跨ぎ FK が全部消え、orders の他ドメイン依存は
   **ItemsPort/UsersPort interface** ただ 1 本になった。port モードでは orders は他ドメインの表に一切触れない。
   join モードに残る `UPDATE items` すら `OrdersRepoPort` の裏——**env 1 個で port に切替でき、usecase/domain は無変更**。
   これがそのまま次段 Lv18 の**抽出の seam**（アダプタ差し替え + DB 移動で live 抽出できる）。
2. **層化は runtime タダ**。latency は mud/Lv13 と同じ（med ~5ms、interface 越しの間接呼び出しは無料）。
   Lv14「modular 化は runtime 無料」の再演——**払うのは設計規律であって性能ではない**。
3. **hexagonal を cargo-cult しない**。usecase 層はオーケストレーションがある orders でこそ価値を持ち、thin CRUD の
   items/users では passthrough を作らない。並行性を持つ不変条件（在庫充足）は純 domain でなく DB の atomic guard に置く。
   **「層を全ドメインに機械的に敷く」のは泥団子の逆の過剰設計**——境界は「切る意味がある所」に引く。

## 払ったコスト / トレードオフ
- **参照整合の移動**: 跨ぎ FK を捨てたので「存在しない item への order」を DB が防がなくなる（アプリ/usecase の責務に）。
  これは分解の本質的コストで、Lv19 の分散 tx 議論に直結。
- **ファイル数増**（domain/usecase/infra/ports/routes）。小さいドメインには過剰になりうる（→ thin CRUD は層を省いて回避）。

## 実装メモ / 罠
- builder 実装 → qa-reviewer で「外部契約バイト互換・依存の向き・join/port が usecase 集約・cache-aside/ASYNC_WRITE 保存」を
  全 endpoint 確認（must-fix 無し）。旧 repo.ts/service.ts と未使用 thin usecase は削除（dead code）。
- latency 計測時 load が 6-8 に跳ねてテールが荒れた（p95 が一時 200ms）。**median を頑健な信号として読む**（med は不変）。
