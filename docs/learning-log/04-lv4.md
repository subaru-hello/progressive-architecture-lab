# Lv4 — 単一DBの壁：コネクション枯渇 → PgBouncer

記録日: 2026-07-08 / マシン: 8 コア / Docker Compose / postgres:16-alpine / PgBouncer 1.25.2

## やったこと
- Lv0-3 で残していた「単一 DB は水平スケールの外」(教訓12)の壁を、実際に負荷で叩いて再現。
- **フェーズA**: api×4 を DB へ直結（各 pool 15）→ `max_connections=20` を食い潰し **7.61% エラー**。
- **フェーズB**: 間に PgBouncer(transaction pooling) を挟む → 同一負荷で **0.00% エラー**。
- 壁の変数（接続数）を切り分けるため、あえて k8s から compose に戻した。

## なぜ compose に戻したか
`max_connections` を絞って観測する実験は、k8s より compose の方が制御・観測が素直。
本番 k8s なら PgBouncer は Deployment/Service になるだけで、考え方は同一。

## フェーズA — 壁の再現（直結）
```console
$ docker compose -f stages/04-db-pooling/docker-compose.wall.yml up --build -d
$ docker exec pal-lv4-wall-db-1 psql -U pal -d pal -tAc "show max_connections;"
20
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8080 grafana/k6 run - < load/db.js
  ✗ http_req_failed  rate=7.61%
    checks_succeeded...: 92.38% 45392 out of 49134
    http_reqs..........: 49134  1227.6/s
    http_req_duration..: p(95)=51.05ms
```
原因を裏取り:
```console
$ docker logs pal-lv4-wall-db-1 2>&1 | grep -c "too many clients"
3751                       # 失敗リクエスト 3742 とほぼ一致
$ docker logs pal-lv4-wall-db-1 2>&1 | grep "too many clients" | tail -1
FATAL:  sorry, too many clients already
```
**算数**: api 4 レプリカ × `PG_POOL_MAX=15` = 最大 60 本の接続要求 ≫ DB の 20 スロット（実質~17）。

## フェーズB — PgBouncer で解く
```console
$ docker compose -f stages/04-db-pooling/docker-compose.wall.yml down
$ docker compose -f stages/04-db-pooling/docker-compose.pgbouncer.yml up --build -d
# 効いている設定（生成 pgbouncer.ini）:
#   pool_mode = transaction / default_pool_size = 15 / max_client_conn = 1000 / auth = scram-sha-256
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8080 grafana/k6 run - < load/db.js
  ✓ http_req_failed  rate=0.00%
    checks_succeeded...: 100.00% 49283 out of 49283
    http_reqs..........: 49283  1231.8/s
    http_req_duration..: p(95)=44.26ms
```
DB は `max_connections=20` のまま。負荷中の DB 実接続を数えると:
```console
$ docker exec pal-lv4-pgbouncer-db-1 psql -U pal -d pal -tAc \
    "select count(*) from pg_stat_activity where usename='pal';"
17     # うち1本は計測用の直結 psql。実質 ~15 = default_pool_size に集約。< 20 なので枯渇しない
$ docker logs pal-lv4-pgbouncer-db-1 2>&1 | grep -c "too many clients"
0
```

## 対比表
| | フェーズA 直結（壁） | フェーズB PgBouncer |
|---|---:|---:|
| エラー率 | **7.61%**（3742 / 49134） | **0.00%**（0 / 49283）✓ |
| スループット | 1227 req/s | 1232 req/s |
| p95 | 51ms | 44ms |
| DB `too many clients` | **3751 件** | **0 件** |
| DB への実接続 | 20 で頭打ち→枯渇 | ~15 に集約 |

## いちばんの気づき（Lv0-1 と別種の壁）
**スループットはほぼ同じ（1227 vs 1232）。** この箱では DB にまだ CPU/IO の余裕があり、
壁は「速度」ではなく**可用性（接続枯渇で 5xx が出る）**だった。
PgBouncer が消したのは**エラー**で、速度を足したのではない。
Lv0-1 の「レプリカで throughput が伸びる」壁とは種類が違う ＝ **スケールの壁は throughput とは限らない**。

## transaction pooling のキモ
`pool_mode=transaction` は、トランザクション単位で server 接続を貸し出して即返させる。
だから 60 本の client 接続が 15 本の server 接続を回し使いできる。
（`session` モードだと接続を掴みっぱなしになり効果が薄い。短い query 多数の Web API は transaction が定石。）
※ transaction モードではセッション状態（`SET`、セッション一時テーブル、prepared statement 等）を跨げない制約がある。今回のアプリは単発 query のみなので影響なし。

## ⚠️ 正直な注記 — リードレプリカを今回やらない理由
読み取りスケール（リードレプリカ）は「実ハードが増える」前提で効く。
同一 Mac 1 台ではレプリカも**同じコアを共有**する（Lv3 の k3d 3ノードと同じ罠）。
接続枯渇は 1 台でも忠実に再現・解消できるので先に PgBouncer を扱った。
リードレプリカ＋read/write 分離は**実ノードが増える環境で Lv5** に回す。

## 気づき（LEARNINGS.md に反映済）
1. **アプリを水平スケールすると、単一 DB の"接続数"が先に枯渇する。** 4×15=60 要求 vs 20 スロット → `too many clients`。
2. **コネクションプーラは client 接続と server 接続を分離する。** PgBouncer が 60 client を 15 server に集約し、同一負荷で 7.61%→0% エラー。
3. **スケールの壁は throughput とは限らない。** 今回の壁は速度でなく可用性（5xx）。プーラは速度でなくエラーを消した。
