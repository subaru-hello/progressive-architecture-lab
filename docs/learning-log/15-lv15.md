# Lv15 — microservices：分解の対価を回収する（network 税・分散データ・障害隔離）

記録日: 2026-07-12 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~3-5

## やったこと
- **プロセスを 3 つに分割** + **DB-per-service**: items-api / orders-api / users-api がそれぞれ専用の
  Postgres（items-db / orders-db / users-db）を所有。前段に Caddy gateway（:8080）を置き path で振り分け。
- `ItemsPort`/`UsersPort` の in-process アダプタを **HTTP アダプタ**に差し替え（`ITEMS_SERVICE_URL` /
  `USERS_SERVICE_URL` を設定）。orders-api は order 作成のたびに **users-api へ認証 / items-api へ在庫**を
  HTTP で呼ぶ。**ドメインコードは Lv13/14 と同一 image**——差し替えたのはアダプタと env だけ。
- DB-per-service なので **orders は items を JOIN できない**。在庫デクリメントは items-db、order INSERT は
  orders-db → **別 DB 跨ぎ**。Lv13 の「1 tx で原子的」が構造的に不可能になる。

## 環境 / セットアップ
```bash
docker compose -f stages/15-microservices/docker-compose.yml up --build -d
# items-api: SERVICE=items / users-api: SERVICE=users
# orders-api: SERVICE=orders, ORDERS_CROSS_CONTEXT=port, ITEMS_SERVICE_URL/USERS_SERVICE_URL 設定
# 各 api は自分専用の DB。gateway(:8080) が /items→items /orders→orders /users→users を振り分け。
```

## コマンドと出力
```console
### 疎通（gateway 経由・別プロセスが応答）###
$ curl -s :8080/items | head -c 80          # instance=f1b9…(items-api)
$ curl -s -XPOST :8080/orders -H 'x-auth-token: demo-token-1' -d '{"itemId":3,"qty":4}'
{"instance":"50f8…","order":{"id":1,"user_id":1,"item_id":3,"qty":4,...}}   # ← orders-api(別プロセス)
$ curl -s :8080/items/3      # items-db の在庫が網越しに減っている
{"instance":"f1b9…","item":{"id":3,...,"stock":999996,...}}                 # 1000000→999996(-4)

### steady 負荷（RATE=50, 30s, Lv13/14 と同条件）###
order_latency: avg=7.29ms med=5.95ms p90=8.14ms p95=10.71ms max=122ms   http_req_failed=0.00%

### blast radius（依存サービス障害の隔離）###
$ docker compose ... stop users-api        # 認証依存を落とす
$ curl :8080/items   -> 200                # items-api は独立 → 生存（隔離）
$ curl -XPOST :8080/orders -> 500          # auth 依存喪失 → 5xx（silent success ではない）
$ docker compose ... start users-api
$ curl -XPOST :8080/orders -> 201          # 復帰

### idle 常駐メモリ ###
items-api 19.6 + users-api 17.9 + orders-api 20.1 + 3×db(~33each=100) + caddy 20 = ~177MiB
```

## 実測データ（Lv13-15 背中合わせ・全 A/B 集約）
| 指標 | Lv13 monolith | Lv14 modular | Lv15 micro |
|---|---|---|---|
| cross-context | 直 SQL JOIN | in-process ポート | **HTTP hop ×2** |
| プロセス数 | 1 | 1 | **3 + gateway** |
| DB | 共有 1 | 共有 1 | **DB-per-service ×3** |
| order_latency **med** | 5.34ms | 5.38ms | **5.95ms** |
| order_latency **p95** | 7.89ms | 8.13ms | **10.71ms** |
| order_latency avg | 5.89ms | 6.03ms | **7.29ms** |
| idle 常駐メモリ | ~53MiB | ~53MiB | **~177MiB（3.3×）** |
| 原子性(在庫↔order) | 1 tx で原子的 | 1 tx で原子的 | **別 DB 跨ぎ=非原子** |
| blast radius | 全滅（プロセス1） | 全滅 | **隔離（/items 生存）** |
| 失敗率(steady) | 0% | 0% | 0% |

## いちばんの気づき — 分解は「in-process の単純さ」を「独立性」と引き換える取引
1. **latency 階段は出るが小さい**: med 5.34 → 5.38 → **5.95**（micro は 2 HTTP hop で +0.6ms）、
   p95 7.89 → 8.13 → **10.71**（+2.8ms）。**monolith ≈ modular < micro** の順は一貫。ただし
   **loopback なので network 税は小さい**——同一ホスト内の HTTP は sub-ms。**実 network（cross-AZ）なら
   各 hop 1-10ms、税は 10-100×に膨らむ**。ここでの +0.6ms は「network 税の下限」と読む。
2. **本当の対価は latency ではなく別のところに出る**（Lv12 と同型: warm 定常は差より他が本体）:
   - **idle 常駐 3.3×**（53 → 177MiB）: プロセスと DB が 3 倍。散発トラフィックほど割高。
   - **原子性の喪失**: DB-per-service で在庫(items-db)と order(orders-db)が別 tx。**在庫を引いた後に
     order INSERT が失敗すると、在庫は減ったまま order が消える**——補償(saga)を書かない限り恒久リーク。
     Lv13 でタダだった原子性が、分解した瞬間に「自分で実装する分散トランザクション問題」に化ける。
   - **blast radius の隔離**（＝分解の見返り）: users-api を落としても /items は 200 継続。
     monolith なら同じ障害でプロセスごと全滅。**障害の粒度を細かくできるのが micro を選ぶ理由**——
     そのために latency 税と分散データの痛みを払う。
3. **HTTP アダプタは失敗を正直に伝える**: 依存 down で `/orders` は 500（`AbortSignal.timeout(2000)` +
   非 2xx を throw）。**silent success にしない**のが blast radius デモを本物にする鍵（Lv2 の
   「502 高速返しで throughput が嘘をつく」の教訓を踏まない）。

## 払ったコスト / トレードオフ
- **運用複雑性**: 3 デプロイ単位・3 DB・gateway・サービス間の契約(`/internal/*`)。監視も 3 面。
- **分散データ整合**: 別 DB 跨ぎの非原子性。本番なら saga / outbox / 冪等リトライが要る（本ラボは
  「痛みを見せる」ために意図的に補償を書いていない）。
- **見返り**: 独立デプロイ・独立スケール（`--scale orders-api=3` で orders だけ増やせる）・障害隔離。

## 実装メモ / 罠
- **Docker-on-Mac 歪み**: loopback HTTP は実 network より速い。latency 階段は「差の向きと一貫性」で読み、
  絶対値は割り引く。box load でテールが荒れる（初回計測で Lv13 p95 が 46ms に跳ねたが load を下げて再測 → 7.89ms）。
  **load<5 の窓で背中合わせ**が必須（Lv12 で確立した手順）。
- **gateway は外向き path だけ公開**。`/internal/*`（cross-context の HTTP 先）は晒さず、サービス間で直接叩く。
- **join+HTTP 誤設定ガード**: `ORDERS_CROSS_CONTEXT=join` かつ `*_SERVICE_URL` 設定は起動時 throw
  （join は items テーブル直参照だが micro の orders-db に items 表は無い＝全 order 500 になる。fail-fast）。

## 次へ（分解軸の総括）
- トポロジ軸（Lv0-12）に続き分解軸（Lv13-15）も一区切り。**結論: 分解の是非は「latency」でなく
  「独立性がその運用コストに見合うか」で決まる**。単一チーム・単一トラフィックなら monolith/modular が正解、
  多チーム並行デプロイ・障害隔離が要る規模で初めて micro の税が黒字化する（Lv12「損益分岐 workload 数」と同型）。
- 未消化: 実 network 上での税の実測（k3d に載せれば hop が現実的に）、saga での原子性回復、独立スケールの数値化。
