# Lv2 — Kubernetes 単一ノード（k3s / k3d）

記録日: 2026-07-06 / マシン: 8 コア / k3d 5.9.0 / k3s v1.35.5

## やったこと
- `k3d` を入れて単一ノードの k3s クラスタを作成。
- 同じアプリを Deployment/Service/Ingress で宣言的にデプロイ（Postgres は PVC で永続化）。
- 自己修復を確認し、**CPU 飽和で Node×k8s が事故る**現象を発見→診断→対処まで。

## ⚠️ 事件: CPU 飽和で 98% エラー
Lv1 と同じ CPU 負荷（cpu=10, 20VU）をかけたら、数字が明らかに異常だった:
```console
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8081 -e CPU=10 grafana/k6 run - < load/cpu.js
    checks_succeeded...: 1.18%  139 out of 11717
    http_req_failed....: 98.81% 11578 out of 11717     # ← 98.8% がエラー
    http_req_duration..: med=1.39ms max=6.18s          # median 1.4ms = 高速エラー返し
    http_reqs..........: 11717  455/s                  # 「高スループット」は錯覚
```
最初は `limits.cpu:"1"` の CFS スロットリングを疑って外したが、それでも 98% エラー。
イベントを見たら真因が出た:
```console
$ kubectl -n pal get events | grep -iE 'probe|unhealthy'
Warning  Unhealthy  pod/api-...  Liveness probe failed: .../health: context deadline exceeded
Warning  Unhealthy  pod/api-...  Readiness probe failed: .../ready: context deadline exceeded
```

### 何が起きたか
1. `cpu=10` の処理が **Node の単一イベントループを塞ぐ**。
2. その間 `/health`(liveness)・`/ready`(readiness) が応答できず **probe がタイムアウト**。
3. readiness 失敗 → k8s が Pod を **Service の endpoints から外す**。liveness 失敗 → **Pod 再起動**。
4. 3 Pod が同時に抜ける → バックエンド消滅 → Traefik が **502 を高速で量産**（だから median 1.4ms・throughput 高く見える）。
5. Lv1(Caddy) には probe が無く、詰まっても endpoints から外れないので捌けていた。**k8s は probe が能動的に Pod を抜くぶん、飽和時に雪崩れる。**

### 対処1: probe を寛容化 + startupProbe 分離
```yaml
startupProbe:  { failureThreshold: 30, periodSeconds: 3 }         # 起動待ちで殺さない
readinessProbe:{ timeoutSeconds: 5, failureThreshold: 3, periodSeconds: 10 }
livenessProbe: { timeoutSeconds: 5, failureThreshold: 6, periodSeconds: 15 }
```
```console
$ docker run ... -e CPU=10 grafana/k6 run - < load/cpu.js
    checks_succeeded...: 95.91%    # 98%エラー → 4%エラーに改善（cascade は止まった）
    http_req_failed....: 4.08%
    http_req_duration..: avg=2.9s p(95)=8.39s     # ただし激遅
    http_reqs..........: 147  6.4/s               # スループット崩壊
```
- **cascade（502連鎖）は止まった**が、今度は 6.4 req/s・p95 8.4s。
- 理由: **単一スレッドが本当に飽和している**以上、probe を緩めても遅延そのものは消えない。

### 現実的な I/O 負荷なら正常
```console
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8081 grafana/k6 run - < load/smoke.js
    checks_succeeded...: 100.00%
    http_req_failed....: 0.00%
    http_req_duration..: p(95)=36.99ms
    http_reqs..........: 2140  60.9/s
```
- 実際の API が主にやる **I/O 待ち（ms=20）なら 0% エラー・p95 37ms で快調**。プラットフォームは正常。
- つまり事故は「意図的に CPU を飽和させた」時だけの話。とはいえ現実にも重い CPU 処理はあり得るので、対処は知っておくべき。

## 結果まとめ
| 負荷 | エラー率 | スループット | p95 |
|---|---:|---:|---:|
| I/O smoke (ms=20) | 0% | 60.9 req/s | 37ms |
| CPU cpu=10（既定 probe） | **98.8%** | (502量産) | — |
| CPU cpu=10（寛容 probe） | 4% | 6.4 req/s | 8.4s |

## 気づき
1. **k8s のメトリクスは「エラー率」を見ないと騙される。** 502 の高速返しは throughput を高く見せる。`http_req_failed`/`checks_succeeded` を必ず確認。
2. **probe は諸刃の剣。** 自己修復の要だが、Node のように単一スレッドが詰まると「健全な Pod を殺す」方向に働き雪崩を起こす。`startupProbe` 分離＋`timeout/failureThreshold` の調整が要る。
3. **緩めても単一スレッド飽和は治らない。** 本当の解は (a) CPU 処理を `worker_threads` に逃してイベントループを空ける、(b) 飽和前に容量を足す（HPA・ノード増）＝ **Lv3**。

## 次へ進む理由（Lv2 の限界）
- レプリカ数が手動固定。**負荷に応じた自動増減が無い**（飽和してから人間が `kubectl scale` では遅い）。
- 1 ノードなのでスケールは**そのノードのコア数が上限**。
- → **Lv3**: 複数ノード + **HPA** で CPU に応じて Pod を自動増減し、複数ノードのコアを使う。飽和事故を「飽和する前に増設」で回避できるか検証する。
