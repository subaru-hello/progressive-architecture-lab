# Lv9 — Batched INSERT：commit 回数 N→1 が drain の律速だった

記録日: 2026-07-11 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine + redis:7-alpine + Node consumer

## やったこと
- **Lv8 の宿題 = write の耐久 throughput 自体を上げる。** Lv8 で分かったのは「キューはバッファであって乗数ではない」、
  持続 throughput は **consumer の drain レート（DB commit 律速）** で決まるということ。単一 consumer は drain し切れず
  depth が上限 100k に張り付き write の半分が 503 で弾かれていた。
- **Lv8 の consumer は XREADGROUP で N 件まとめて読んでも、`for` ループで 1件ずつ INSERT + commit していた**
  （＝読みはバッチ、書きは per-row）。これが drain 律速の正体。
- Lv9 = 読んだバッチを **1本の multi-row INSERT（`INSERT INTO items(name) VALUES ($1),($2),...`）＝ 1 commit** に束ね、
  XDEL/XACK も ioredis pipeline で一括送信。commit 回数を N→1 に落とす。**consumer 数は 1 に固定**して
  「レプリカを増やさず吞吐量を上げられるか」を測る。
- アプリ変更（後方互換: `BATCH_WRITES` 未設定なら Lv8 と完全同一の per-row 挙動）:
  - `app/src/consumer.ts`: `BATCH_WRITES=1` で `processBatch()` を使う経路を追加。`WRITE_BATCH_SIZE`（既定 100）を
    XREADGROUP の COUNT に。valid を 1本の multi-row INSERT に束ね、成功時のみ **XDEL 全件 → XACK 全件** を pipeline 送信。
    malformed（`name` 欠落）は INSERT から除外し個別 XDEL/XACK で破棄（バッチをブロックしない）。
    Histogram `write_queue_batch_size`（1 commit あたりの行数）を追加。`consumeLoop`/`autoclaimLoop` の両方がバッチ化。

## 実測（write-heavy 90%write @400VU、per-row↔batched 背中合わせ A/B、consumer 1 固定）
```console
### Lv8 per-row（baseline）###
ピーク depth  : 100,059（上限 100k 張り付き）
write 503     : 183,915（54.84% が背圧で失敗）
drain 実績    : 45,153 行 / 平均 commit lag ~54.8s
バッチサイズ  : 1 行/commit
write p95     : 67.7ms（← 受理された 45% のみの値）

### Lv9 batched ###
ピーク depth  : 283（詰まらない。timeline 0→7→56→104→52→17→34→180→0）
write 503     : 0（0.00%）
drain 実績    : 288,969 行（6.4×）/ 平均 commit lag ~20.1ms（≈2,700× 低）
バッチサイズ  : 平均 32.9 行/commit（8,776 commit で 288,969 行）
write p95     : 75.5ms（全 write を受理しての値）
```

## いちばんの気づき — 律速は「INSERT の回数」であって「INSERT する行数」ではなかった
1. **commit を N→1 に束ねただけで drain throughput が 6.4×。** consumer は 1 のまま。Lv8 で
   「drain を上げるレバーは consumer 並列化 or バッチ化」と書いたうちの**バッチ化を単独で効かせた**結果、
   45,153 行 → 288,969 行に増えた。DB に流し込む行数は同じでも、**commit（fsync）の回数**が 1/33 になった。
2. **飽和キューが「詰まる前に捌ける」キューに変わった。** per-row は depth が上限 100k に張り付き write の
   54.84% を 503 で弾いていたが、batched は depth がほぼ 0 のまま（ピーク 283）流れ **503 ゼロ**。
   同じ producer 速度に対し drain が追い越したので、depth も lag（~55s→~20ms）も崩壊した。
3. **なぜ効くか = commit のコストは「行数」でなく「回数」に強く比例する。** 1 commit あたり fsync/WAL flush/
   トランザクションのオーバーヘッドがほぼ固定費。per-row はこの固定費を 1 行ごとに払う。multi-row INSERT は
   33 行分の固定費を 1 回にまとめる。enqueue(XADD) が INSERT より桁違いに軽い（Lv8 教訓27）のと同じ構図が
   「INSERT の中」にもあった——**律速の単位は行ではなく commit**。
4. **write p95 の見かけは逆転して見えるが罠。** per-row 67.7ms < batched 75.5ms だが、per-row の値は
   **受理された 45% のみ**（残り 55% は 503 即弾きで write_latency に入らない）。batched は 100% 受理しての 75.5ms。
   「全部受けて 75ms」と「半分捨てて 68ms」を並べているだけ。**エラー率を見ないと latency に騙される**（Lv3 の教訓の再演）。

## 払ったコスト / トレードオフ
- **poison batch の増幅係数 = batch size。** per-row なら INSERT 失敗 1 件の巻き添えは自身 1 件だが、batched では
  1回の INSERT 失敗で同一バッチ最大 `WRITE_BATCH_SIZE`(=100) 件の正常行が XDEL/XACK されず PEL に残り、
  次の XAUTOCLAIM でまた 100 件まとめて再 INSERT を試みる。DB デッドロック/`too many connections`/statement timeout は
  「DB ダウン時のみ」の想定より現実に起きうる。**耐障害性の粒度をバッチサイズぶん粗くする**のがバッチ化の代償。
- **shutdown ドレインの取りこぼしも batch size ぶん増幅する。** INSERT 済み↔pipeline XDEL/XACK 前で打ち切ると
  宙吊り（再配送→重複）が最大 batch size 件に。Lv9 では consume/autoclaim 両ループを shutdown で await する対策を入れた。
- **at-least-once は不変。** INSERT 失敗時はバッチ全 id を XDEL/XACK せず PEL に残す＝消失なし。重複はしうる（冪等化しない＝教材）。

## 実装の罠（qa レビューで潰した）
1. **ioredis の `pipeline.exec()` は個別コマンドのエラーでは reject せず `[err, result]` の配列で resolve する**
   （reject は接続断など transport レベルのみ）。素朴に `await pipe.exec()` を try/catch するだけだと、
   一部の XDEL/XACK だけが失敗しても catch は発火せず、「XACK 済みだが XDEL 未」＝ XLEN 恒久リーク（Lv8 の肝の不変条件）や
   60s 後の重複再 INSERT が **warn すら出ず不可視**で進む。→ exec の戻り値配列を走査して失敗件数を検出・ログ化。
   per-row 経路（`ackAndDelete`）は逐次 await なので各コマンドのエラーが catch される——この可視化は batch 経路だけに必要だった。
2. **autoclaim の 10s sleep を素朴に shutdown で await すると毎回最大 10s stall**（docker stop grace を食い潰し SIGKILL）。
   → `running=false` で早期に抜ける中断可能 sleep（250ms 刻み）に差し替えてから await。
3. **空バッチ（全件 malformed）で `VALUES ` 単独の不正 SQL を投げない**よう、valid 0 件は INSERT 前に早期 return。

## 次の壁（Lv10 への引き）
- **バッチ化は commit 固定費を薄めたが、DB の物理 write 帯域そのものは増やしていない。** さらに負荷を上げれば、
  今度は 1 consumer の multi-row INSERT でも drain が追いつかなくなる（COPY で更に固定費を削るか、consumer を並列化するか、
  write を分割＝partition/shard するか）。バッチ化は「同じ DB を効率よく使う」施策で、DB の容量壁は依然そこにある。
