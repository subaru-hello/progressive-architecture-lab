# Lv5 — リードレプリカ + read/write 分離：壁は「消える」のでなく「移動する」

記録日: 2026-07-09 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine（ストリーミングレプリケーション）

## やったこと
- Lv4 の宿題（「リードレプリカは実ノードが増える環境で Lv5 に回す」）を実測。
- アプリを **2 pool 化**：`writePool`→primary（INSERT/DDL）、`readPool`→replica（SELECT）。
  - 後方互換: `DATABASE_URL_RO` 未設定なら `DATABASE_URL` に fallback → Lv0-4 の compose はそのまま動く。
- **フェーズA（壁）**: 分離なし。read も write も single primary に集中（`docker-compose.wall.yml`）。
- **フェーズB（分離）**: primary + standby のストリーミングレプリケーション、read を replica に逃がす（`docker-compose.replica.yml`）。
- 負荷 `load/read-heavy.js`（read 95% / write 5%、write と read の p95 を tag+Trend で分離、`PEAK_VUS` で負荷段階を切替）。
- **主指標は write の p95**（total throughput ではない ← コア共有下では throughput は multiplex しないため。教訓6・11 の延長）。

## レプリケーションの立て方（公式イメージ手組み）
Lv4 までの official image 路線を維持。bitnami の隠蔽層を使わず「仕組み」を残す。
- **primary**: `command` で `wal_level=replica / max_wal_senders=10 / hot_standby=on / wal_keep_size=64MB`。
  init スクリプトで replication ロール作成 + `pg_hba.conf` に `host replication` 行追加。
- **replica**: entrypoint ラッパーで primary healthy を待ち、PGDATA が空なら `pg_basebackup -R`
  （`-R` が `standby.signal` と `primary_conninfo` を自動生成）。公式 entrypoint を上書きするので
  **`gosu postgres` で root から降格**（postgres は root 起動を拒否）＋ PGDATA の所有権を postgres に。
- PGDATA は anonymous volume ＝毎回クリーンに basebackup し直す（教材の事故防止）。

成立確認:
```console
$ docker compose -f stages/05-read-replica/docker-compose.replica.yml up --build -d
$ docker compose -f ... exec -T db-primary psql -U pal -d pal -c \
    "SELECT client_addr, state, sync_state FROM pg_stat_replication;"
 client_addr | state     | sync_state
-------------+-----------+------------
 172.20.0.3  | streaming | async          # replica が streaming で繋がっている
$ docker compose -f ... exec -T db-replica psql -U pal -d pal -tAc "SELECT pg_is_in_recovery();"
 t                                          # replica は standby（recovery 中）
$ curl -s localhost:8080/replication
{"instance":"...","lag_seconds":1.07}       # ラグを数値で観測できる
```

## フェーズA vs B — moderate 負荷（150 VU, read95/write5, 45s）
```console
# 壁（分離なし・primary に read+write 集中）
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8080 grafana/k6 run - < load/read-heavy.js
  write_latency p(95)=40.17ms / read p(95)=33.45ms / http_reqs=1976.9/s / http_req_failed=0.00%
# 分離（read を replica へ）
$ docker run --rm -i -e BASE_URL=http://host.docker.internal:8080 grafana/k6 run - < load/read-heavy.js
  write_latency p(95)=41.31ms / read p(95)=36.21ms / http_reqs=1948.6/s / http_req_failed=0.00%
```
負荷中の primary を `docker stats` + `pg_stat_activity` でサンプリング:

| 150 VU | 壁（分離なし） | 分離（replica） |
|---|---:|---:|
| write p95 | 40.2ms | 41.3ms |
| read p95 | 33.5ms | 36.2ms |
| throughput | 1977 req/s | 1949 req/s |
| **primary CPU** | **~65–79%** | **~8%** |
| **primary 接続数** | **~56** | **~15** |

→ **レイテンシ・スループットはほぼ同一。** だが **primary の CPU を ~85%、接続を ~73% オフロード**した。
この負荷帯では primary はまだ飽和しておらず（~79%）、壁は「速度」でなく **primary の余力** に現れた。

## フェーズA vs B — 高負荷で primary を飽和させる（400 VU）
`PEAK_VUS=400` で壁側 primary を 1 コア超えまで押し込む:

| 400 VU | 壁（分離なし） | 分離（replica） |
|---|---:|---:|
| write p95 | **145.3ms** | **147.3ms** |
| read p95 | 137.0ms | 158.6ms |
| throughput | 3121 req/s | 2794 req/s |
| primary CPU | **~110–132%**（飽和） | **~17–27%**（余裕） |
| replica CPU | —（無し） | **~110–122%**（飽和） |
| primary 接続数 | ~67 | ~28 |
| レプリケーションラグ（負荷中） | — | ~0.02–0.07s |

→ **write p95 は 145 ≈ 147ms で改善しなかった。** むしろ read p95 と throughput は分離側が**悪化**（158 vs 137ms / 2794 vs 3121 req/s）。
primary の CPU は 130%→22% に劇的に下がったが、その read 負荷は**同じ箱の replica に移り 120% で飽和**しただけ。

## いちばんの気づき — 壁は消えず「移動」した
**単一ホストでは read/write 分離はレイテンシもスループットも改善しない。** read の CPU 負荷が
同じ共有コア上の replica へ移るだけで、箱全体は依然 CPU 律速。レプリケーション overhead のぶん
むしろ僅かに悪化する。primary の CPU 130%→22% オフロードは**本物**だが、その果実
（read をさばく容量が増える）は **replica が別ハード（別コア）にある時だけ**効く。

これは Lv3 の「k3d 複数ノードは同じコアを共有＝実コアは増えない」（教訓11）が、
今度は **DB 層**で再来しただけ。分離という**パターンは正しい**——効くかどうかは**トポロジ（実ハードが分かれているか）**で決まる。

## レプリケーションラグの正直な注記
- **負荷中**のラグは ~0.02–0.07s（20–70ms）と小さいが非ゼロ → `POST` 直後の `GET` が
  新規行を取りこぼす瞬間がありうる（read-after-write の不整合）。今回は分離をあえてフォールバック無しで実装し、この性質を隠さない。
- **アイドル時**は `/replication` のラグ値が青天井に増える（15s 等）。これは
  `now() - pg_last_xact_replay_timestamp()` が「最後に replay した時刻からの経過」なので、
  新規トランザクションが無いと単に古びるだけ ＝ **遅延ではなくアーティファクト**。ラグは負荷を流している最中に測らないと意味がない。

## 次へ進む理由
- 分離の真価（read の水平スケール）を測るには **replica を別マシン**に置く必要がある（自宅なら 2 台目の Linux、実ノードが増える環境）。
- read 落下時に primary へ寄せるフォールバックや、複数 replica を束ねる L4 LB（HAProxy）は、
  実ノードが分かれてから足すのが素直（単一箱では「見せかけのスケールアウト」になる）。

## 気づき（LEARNINGS.md に反映済）
1. **read/write 分離は primary を確実にオフロードする**（CPU 130%→22%、接続 67→28）。だが**単一ホストでは client 視点のレイテンシは改善しない**——read 負荷が同じコアの replica に移るだけ。
2. **壁は消えず移動する。** 分離という*パターン*は正しいが、*効果*はトポロジ（replica が別ハードか）で決まる。Lv3 のコア共有の壁が DB 層で再来。
3. **レプリケーションラグは負荷中に測る。** アイドル時の `now()-replay_ts` は青天井に増えるアーティファクト。負荷中の実ラグは ~20–70ms（read-after-write の不整合源）。
