# Lv19 — 分散トランザクション：none / 2PC / saga —— タダの分散tx は無い

記録日: 2026-07-13 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~5-9

## やったこと
- Lv18 で items が別サービス+別DBになった世界。POST /orders は **在庫デクリメント(items-db)+order INSERT(orders-db)の
  2DB 跨ぎ**。orders app を coordinator として、`ORDER_TX_MODE=none|2pc|saga` の3方式を実装し、**障害注入**で比較。
- **none**: 無調停(decrement→insert)。
- **2pc（本物）**: app オーケストレートの 2 相コミット。items-db(items-service 経由)と orders-db の各々で
  `BEGIN;…;PREPARE TRANSACTION 'gid'` → 両 prepared 後に `COMMIT PREPARED`(失敗で `ROLLBACK PREPARED`)。
  両 PG に `max_prepared_transactions=50`。
- **saga（orchestration + 補償 + recovery poller）**: 予約(items-service, **冪等 reservation id**)→order INSERT→
  saga_log completed、失敗で**補償**(release)。orders-db の `saga_log` を 2s 間隔の poller が走査し、`reserved` で滞留した
  saga(クラッシュで宙吊り)を `compensating`→release→`compensated` に自己回復。
- 障害注入 `FAULT_POINT`: `after-first-write`(none: decrement 後 / saga: reserve 後に throw)・`after-prepare-all`(2pc:
  両 prepare 後・commit 前に throw)。

## コマンドと出力
```console
### happy path latency（1 VU 逐次・衝突なし・全 0%）###
none : order med 3.42ms p95 12.16ms   （HTTP decrement 1 + local insert 1）
2pc  : order med 5.46ms p95 11.89ms   （2 prepare + 2 commit = ~4 往復 + 各相 fsync → 最遅）
saga : order med 5.04ms p95 10.85ms   （reserve + insert + saga_log 更新）

### none + fault(after-first-write) → 恒久リーク ###
before: items stock_sum=49999816  orders=182
after : items stock_sum=49999796  orders=182      # stock -20・order 0件増 = 在庫だけ消える不整合

### 2pc + fault(after-prepare-all) → in-doubt（両DBに prepared・行ロック保持）###
prepared xacts: items-db=10  db=10                # commit も rollback もされず宙吊り
→ 手動 ROLLBACK PREPARED が要る（DBA 介入）。放置すると行ロックで後続がブロック(可用性低下)。

### saga + fault(after-first-write) → 補償で結果整合（ロック無し）###
t0(直後): stock_sum=49999786  saga=[completed|61 reserved|10]   # reserve で -10・saga 滞留
~6s後   : stock_sum=49999796  saga=[completed|61 compensated|10] # poller が補償→stock 復旧
prepared xacts: items-db=0                                       # saga はロックを一切残さない
```

## 実測データ（3方式の比較）
| 指標 | none | 2pc（本物 PG prepared-tx） | saga（orchestration+outbox） |
|---|---|---|---|
| happy order_latency med | **3.42ms** | 5.46ms（最遅） | 5.04ms |
| 障害注入時の一貫性 | **恒久リーク**（在庫 -20・order 0） | **in-doubt**（両DB prepared 宙吊り） | **結果整合**（補償で自己回復） |
| ロック/ブロッキング | 無し | **行ロック保持**（解決まで） | **無し** |
| 障害からの回復 | 無し（壊れたまま） | **手動**(ROLLBACK PREPARED/DBA) | **自動**(poller ~6s) |
| 中間状態の可視性 | — | 不可視(prepared は隠れる) | **可視**(reserved が見える) |
| 実装複雑度 | 最小 | 中(prepare/commit/rollback 契約) | 大(冪等鍵+補償+outbox poller) |

## いちばんの気づき — タダの分散tx は無い。どのプロパティを諦めるかの選択
1. **none は速いが無防備**。med 3.42ms で最速だが、障害で「在庫だけ減って order 消失」の恒久リーク。
   Lv15 教訓34「原子性が自前の分散tx問題に化ける」が、放置するとこうなる、の実演。
2. **2PC は原子性(C)を「ブロッキング + coordinator 結合 + in-doubt 運用」で買う（可用性 A を捨てる）**。
   happy path は正しく all-or-nothing だが最遅(2 相 × 往復 × fsync)。**coordinator が両 prepare 後に死ぬと
   両DBに prepared txn が宙吊り**し、行ロックを握ったまま手動解決(ROLLBACK PREPARED)まで後続をブロックする。
   分散システムで 2PC が嫌われる核心＝この in-doubt・ブロッキング。
3. **saga は可用性(A)を「中間状態の可視化 + 補償・冪等の実装コスト」で買う（分離性 I を捨てる）**。
   ロックを一切握らず、クラッシュしても poller が補償して結果整合に収束(~6s)。だが「予約済だが未 confirm」の
   **中間状態が外から見える**(在庫は引かれたが order はまだ)、かつ補償ロジック・冪等鍵・outbox/poller を自分で
   書く必要がある。整合は「即時」でなく「最終的」。
4. **= CAP/PACELC がアプリ層に降りてくる**。2DB 跨ぎの一貫性は「原子性(2PC)か可用性(saga)か」の二択で、
   タダの正解は無い。**多くの現代システムが saga を選ぶのは、可用性とスケールを優先し I を諦める判断**
   （2PC のブロッキングは高負荷で雪だるま化する——下記）。

## 払ったコスト / トレードオフ・罠
- **実装中に見つけた本物のバグ（2PC の脆さの実地教材）**: commit の HTTP 呼び出しが「空 body + `Content-Type:
  application/json`」で Fastify に **400** され → prepared txn が commit されず **orphan 化** → 行ロックが溜まり
  後続が全部 timeout → **orphan 雪だるま**で全滅。**2PC は「commit phase の 1 つの失敗」が in-doubt を生み、
  それが伝播する**という脆さそのものを、バグが実演してしまった(修正: 空 body 時は Content-Type を付けない)。
- **2PC は高競合で崩れる**: RATE=30 の並行負荷では、prepared txn の行ロックが衝突し prepare が timeout、
  abandon された prepared が `max_prepared_transactions` を食い潰して雪だるま化(実測: 全滅)。happy latency は
  低競合(逐次)でしか綺麗に測れなかった。**2PC はスループットと相性が悪い**。
- **saga の非冪等リトライ**: reserve/release は reservation id で冪等化必須(でないと二重デクリメント)。
  qa-reviewer が「order insert 後・completed マーク前にクラッシュ → poller が既存 order を誤補償(二重支払い)」の
  窓を指摘——非原子な cross-DB saga の本質的ハザードで、これも「タダでない」ことの一部。
- 計測は load<5 の窓が取りづらく(5-9)、happy latency は median を頑健信号に。障害注入は本物の throw で。

  seam×手順の積**(Lv18)、そして切った後に残る**cross-DB 原子性は 2PC(C) か saga(A) の二択で買うしかない**(Lv19)。
