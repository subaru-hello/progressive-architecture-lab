# Lv16 — 泥団子(big ball of mud) monolith と「動かせなさ」：引っ越しやすさ＝結合度

記録日: 2026-07-13 / マシン: 8 コア Mac / Docker Compose / 計測時 load avg ~4

## やったこと
- 分解の**完成形**（Lv13-15）に続き、**「引っ越しの過程」軸**に着手。まず出発点として、故意に絡ませた
  **本物の泥団子** `app/src/mud/` を新設（`ARCH=mud` で mount。未設定=`hex` で従来どおり → Lv0-15 は不変）。外部 HTTP 契約は
  hex と同一（`/items` `/orders` …）で、**違うのは内部構造だけ**。
- **故意の cross-domain 結合**（＝引っ越しを阻む結合を意図的に作る）:
  - **共有トランザクション**: `POST /orders` が 1 BEGIN/COMMIT で **items 在庫 + orders + users.last_order_at の3テーブルを横断**（handler 直書き・repo/port 無し）。
  - **cross-domain JOIN**: `GET /orders` が `orders JOIN items JOIN users` で item_name/user_name を埋める。
  - **跨ぎ FK**: `orders.item_id REFERENCES items(id)` / `orders.user_id REFERENCES users(id)`。
  - logic は handler 直埋め・schema は全ドメイン同居の1スキーマ。

docker compose -f stages/16-mud/docker-compose.yml up --build -d   # ARCH=mud, SERVICE=all, SEED_DEMO=1

## コマンドと出力
```console
### 疎通（結合の証拠）###
$ curl -s -XPOST :3000/orders -H 'x-auth-token: demo-token-1' -d '{"itemId":1,"qty":2}'
{"order":{"id":1,"user_id":1,"item_id":1,"qty":2,...}}
$ curl -s :3000/orders          # JOIN で item_name/user_name まで埋まる
{"orders":[{"id":1,"qty":2,...,"item_name":"demo-item-1","user_name":"Alice"}]}
$ psql> SELECT id,name,last_order_at IS NOT NULL AS ordered FROM users;
  1 | Alice | t        # ← 同一 tx で users.last_order_at が更新された = 3テーブル共有 tx の証拠
  2 | Bob   | f

### 結合度カウント（静的・migratability の代理指標）###
cross-domain 共有 tx : 1（POST /orders が items+orders+users を横断）
cross-domain JOIN    : 1（GET /orders）
跨ぎ FK              : 2（orders→items, orders→users）

### 実演A: 跨ぎ FK が items の除去/移動を物理的に拒否 ###
$ psql> DROP TABLE items;
ERROR:  cannot drop table items because other objects depend on it
DETAIL:  constraint orders_item_id_fkey on table orders depends on table items

### 実演B: items を「別DBへ引っ越した」状況を再現 → 直SQL/JOIN が即死 ###
$ psql> ALTER TABLE orders DROP CONSTRAINT orders_item_id_fkey; ALTER TABLE items RENAME TO items_relocated;
$ curl POST /orders  -> 500      # UPDATE items が「relation items が無い」で失敗
$ curl GET  /orders  -> 500      # JOIN items が失敗

### k6 でダウンタイムを定量化（RATE=40, 15s）###
baseline(mud 正常) : order_latency med=5.85ms p95=15.81ms  http_req_failed=0.00%
broken(items 移動後): http_req_failed=100.00%(843/843)  order_create_5xx=601(全order 500)  checks=0%
```

## 実測データ
| 指標 | 値 |
|---|---|
| 結合度: 共有 tx / JOIN / 跨ぎ FK | **1 / 1 / 2** |
| baseline order_latency med / p95 | 5.85ms / 15.81ms（hex Lv13 5.34ms とほぼ同じ＝mud は遅くない） |
| baseline http_req_failed | 0.00% |
| **naive 引っ越し中の http_req_failed** | **100.00%（843/843・全 order 5xx）** |
| items 除去の可否 | **不可**（跨ぎ FK が DROP を拒否） |

## いちばんの気づき — 「引っ越しやすさ」は monolith か micro かでなく、境界の結合度で決まる
1. **mud は遅くない・壊れてもいない・ただ動かせない**。baseline は latency も error 率も hex と遜色ない（med 5.85ms・0%）。
   問題は性能でなく**移行可能性**——items を別 DB に移そうとした瞬間、跨ぎ FK が分割を拒否し（実演A）、直 SQL/JOIN が
   items 不在で即死する（実演B）。**naive にやると 100% ダウン**（843/843 が 5xx）。
2. **結合こそが「引っ越しを阻む重り」**。共有 tx(1)・cross-domain JOIN(1)・跨ぎ FK(2) の一つ一つが、新 DB を跨げない縫い目。
   これらを数える＝migratability を測ること。**monolith だから移せないのではなく、境界が絡んでいるから移せない**。
3. **綺麗にするのが移行の本体**。「まず綺麗にして、切る場所に seam を仕込む」——それをやってから初めて live で移せる。
   次段 Lv17 でこの結合を hexagonal に解きほぐし、結合度が seam 1 本に落ちることを測り、Lv18 で**同じ引っ越しが今度は
   ダウンタイムゼロで通る**ことを示す（Lv16 の 100% ダウンと直接対比）。

## 払ったコスト / トレードオフ
- mud は書くのは速い・読むのも（小さいうちは）速い。**代償は「変更・移行の局所化ができない」**——items を触ると orders が、
  schema を割ろうとすると FK が、全部が芋づるで付いてくる。この「安さ」が後で移行コストとして牙をむく。

## 実装メモ / 罠
- **ARCH トグル**: `ARCH=mud` は mud routes + 共有スキーマのみ mount。未設定/`hex` は従来の hex path をバイト等価で実行
  （qa-reviewer で hex 回帰なしを確認）。`ARCH=mud` + `*_SERVICE_URL` は起動時 throw（**mud には seam が無く config では割れない**、を明示）。
- **実演B は破壊的**（FK 外し + rename）。使い捨て DB 前提で、demo 後は `down -v` で破棄。
- baseline latency は load<5 の窓で採取（Docker-on-Mac 歪みは相対で読む）。

## 次へ（Lv17）
- mud を **DB/Domain/Usecase/Infra の hexagonal(ports & adapters)** に解きほぐす。cross-domain JOIN → ItemsPort 呼び出し、
  跨ぎ FK + 共有 tx → port + ドメイン別 tx、logic → usecase、SQL → infra。**結合度カウントが seam 1 本に激減**し、
  latency は不変（層化の runtime コスト 0）であることを測る。教訓: **hexagonal 境界＝事前に引いた移行の切断線**。
