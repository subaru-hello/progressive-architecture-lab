# Lv8 — write-behind 非同期キュー：バッファであって乗数ではない

記録日: 2026-07-11 / マシン: 8 コア Mac / Docker Compose / postgres:16-alpine + redis:7-alpine + Node consumer

## やったこと
- **本丸 = write のスケール。** read は Lv6 cache で、CPU 律速は Lv7 worker で解いたが、**write はどちらも助けない**
  （cache は write を無効化するだけ、worker は CPU を逃すだけ）。write は常に DB commit（fsync）待ちで律速される。
- `POST /items` を **Redis Stream（`items:writes`）に XADD して 202 即返し**、別プロセスの **consumer** が
  DB commit 律速で淡々と drain する write-behind を実装。client 向け write を DB commit から切り離す。
- アプリ変更（後方互換: `ASYNC_WRITE` 未設定なら現状の同期 INSERT+201。Lv6 `REDIS_URL`/Lv7 `WORKER_POOL_SIZE` と同型）:
  - `app/src/server.ts`: `ASYNC_WRITE` 有効時 `POST` は XADD→**202 `{status:queued, stream_id}`**（DB id 採番せず・cache DEL もしない）。
    背圧 `XLEN >= WRITE_QUEUE_MAX` で 503。Redis 障害の XADD 失敗も 503（**同期フォールバックしない**＝キューは write の本線）。
    `ASYNC_WRITE && !REDIS_URL` は起動時 fail-fast。prom Gauge `write_queue_depth`（XLEN）。
  - `app/src/consumer.ts`（新規・別サービス・同イメージを `node dist/consumer.js` で起動）: XREADGROUP→INSERT→
    **XDEL→XACK**→commit lag observe→cache DEL。at-least-once（XAUTOCLAIM で stale PEL 回収、再配送で重複しうる＝教材）。
    Histogram `write_queue_commit_lag_seconds`。graceful drain（in-flight バッチ完了を await）。

## 実測（write-heavy 90%write、wall↔queue 背中合わせ A/B）
```console
### 低負荷 PEAK_VUS=100（≒1,400 write/s、DB 飽和せず）###
wall  : write p95=13.7ms  p50=3.8ms  read p95=11.2ms  1594 iter/s  (depth 0, lag 0)
queue : write p95=20.7ms  p50=3.7ms  read p95=24.0ms  1550 iter/s  depth 0→41,562  commit lag ~19s
### 高負荷 PEAK_VUS=400（≒3,100+ write/s、DB commit 飽和）###
wall     : write p95=120.1ms  p50=40.0ms  3486 iter/s  背圧0
queue c1 : write p95=72.1ms(受理分)  depth 100k 張り付き  背圧 152,076(51%)  3978 iter/s
queue c3 : write p95=141.7ms※  depth 100k 張り付き  背圧 80,824(35%)  3066 iter/s   ※load28 の自己競合ノイズ
```

## いちばんの気づき — 単純な「速くなる」話ではなかった
1. **DB が飽和しない負荷（100VU）では write-behind は「負け」。** 同期 INSERT p95 13.7ms に対し queue は 20.7ms
   （XADD の 1 ホップぶん遅い）。**4 つの api が同期 INSERT を 40 コネクションで並列に捌けるうちは DB は詰まらず**、
   キューは overhead でしかない。しかも depth が 0→41,562 に単調増加し commit lag ~19s の**負債**だけ残る。
2. **DB が飽和する負荷（400VU）で初めて 202 即返しが効く。** 同期 wall は write p95 120ms まで崩れるが、
   queue は受理された write の p95 72ms。**キューはバーストを吸収する（ramp 中 depth=0、sustained で初めて溜まる）。**
3. **だがキューは「バッファであって乗数ではない」。** 持続的な耐久 throughput = **consumer の drain レート
   （DB commit 律速）**で、enqueue の速さとは無関係。単一 consumer は drain し切れず depth は上限 100k に張り付き、
   **write の 51% が 503 で弾かれた**。上限が無ければ depth は際限なく伸びて OOM——どちらにせよ壁は消えていない。
4. **耐久 throughput を上げる本当のレバーは「キュー追加」でなく「consumer の並列化/バッチ化」。**
   consumer を 1→3 に増やすと背圧が 152k→81k（失敗 51%→35%）に減った＝ drain はほぼ線形にスケール。
   だが enqueue(XADD) は INSERT より桁違いに軽いので、**producer は固定数の consumer を必ず抜く**
   （depth の立ち上がりは consumer を増やしても止まらなかった）。drain 容量を sustained write レートに合わせるのが本質。

## 払ったコスト（202 ≠ 永続化）
- **耐久性**: 201 は永続化済み、**202 は受理のみ**。XADD 前に api が落ちれば消える。AOF off なら Redis 再起動で未 drain 分が丸ごと消える。
- **結果整合性**: enqueue→commit lag（高負荷で平均 ~19s）ぶん GET /items に現れない。Lv6 の cache stale window がさらに広がる。
- **at-least-once**: INSERT 後 XACK 前クラッシュ→再配送で重複 INSERT。冪等化しないと at-least-once は at-least-some-duplicates。

## 実装の罠（qa レビューで潰した）
1. **XACK→XDEL の順は XLEN を恒久リークさせ常時 503 に陥る。** XACK 先行だと PEL から外れ、XACK↔XDEL 間で
   死んだエントリが Stream に残り XREADGROUP('>') も XAUTOCLAIM(PEL のみ) も拾えず、depth が単調増加＝背圧が壊れる。
   **XDEL→XACK 順**にすれば削除済み PEL を XAUTOCLAIM が自己 evict し depth 不変量が自己修復する。
2. **consumer を host ポート束縛(`ports`)にすると `--scale consumer=2` がポート衝突で起動不能。** `expose` にする。
3. **`redis.connect()` の握り潰しは fail-fast を殺す。** Redis 未達でも /ready は green のまま全 POST 503＝write-dead な緑。
   「既接続」の良性 reject だけ握り、本当の失敗は fail-fast。

## 次へ進む理由
- write の耐久 throughput は結局 DB commit と consumer 並列度で決まる。次の手は **INSERT のバッチ化（multi-row COPY）**
  や **partition/shard で commit を分散**。write-behind は「latency を切り離す」施策で「throughput を増やす」施策とは直交する。

## 気づき（LEARNINGS.md に反映済）
1. **write-behind はバッファであって乗数ではない。** 202 は latency を DB commit から切り離すが、耐久 throughput は
   consumer の drain レート律速のまま。sustained enqueue > drain なら depth 無限増加 or 503 背圧——壁は消えない。
2. **キューは DB が飽和して初めて勝つ。** 低負荷（4 api が並列 INSERT で捌ける）では XADD ぶん overhead で「負け」、
   depth/lag の負債だけ残る。最適化の効果は「ボトルネックが本当にそこにあるか」で正負が反転する（Lv5/6 と同系譜）。
3. **enqueue は drain より桁違いに軽いので producer は固定 consumer を必ず抜く。** 耐久 throughput を上げるレバーは
   キュー追加でなく consumer の並列化/バッチ化。consumer 1→3 で背圧 152k→81k（drain はほぼ線形）。
