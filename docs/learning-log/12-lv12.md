# Lv12 — scale-to-zero(serverless風) ↔ always-on：idle コストを cold-start latency に変換する

記録日: 2026-07-12 / マシン: 8 コア Mac / k3d(k3s) 3ノード + KEDA http-add-on / 計測時 load avg ~4-5

## やったこと
- ユーザーの「serverless→EKS 移行を測りたい」を、ラボ憲章（**AWS 不使用・実費ゼロ・k3d だけ**）を守って
  「**scale-to-zero(serverless の肝) → always-on k8s**」のローカル再現に翻訳。EKS は使わない。
  ※ compose→k8s→HPA の移行は Lv2-3 で既済。未着手だった "scale-to-zero" をここで足す。
- **KEDA + http-add-on** を採用（Knative/OpenFaaS を却下）。理由: 既存の普通の Fastify コンテナ（app 不変）を
  そのまま min=0 まで落とせ、Lv3 の Deployment/Service を流用でき A/B の変数が「下限台数 0 vs 2」に純化。
  Mac Docker VM で最軽量。interceptor がリクエストをホールドするので cold-start が「503 でなく遅い 200」で測れる。
- 背中合わせ: **always-on(replicas=2 固定)** ↔ **scale-to-zero(HTTPScaledObject min=0)** を同一クラスタ・同一
  image で制御面だけ差し替えて比較。

## 環境 / セットアップ
```bash
k3d cluster create pal --servers 1 --agents 2 --port "8081:80@loadbalancer"
docker tag pal-api:lv3 pal-api:lv12 && k3d image import pal-api:lv12 -c pal   # app は Lv3 と同一
helm install keda kedacore/keda -n keda --create-namespace --wait
helm install http-add-on kedacore/keda-add-ons-http -n keda --wait
kubectl apply -f stages/12-scale-to-zero/k8s/   # 00,10,20,40,30
```
- interceptor Service = `keda-add-ons-http-interceptor-proxy:8080`(ns keda)。Ingress backend をこれに向ける
  （Ingress は同一 ns 必須なので keda ns に置く）。`HTTPScaledObject.hosts=[pal.localhost]` と Ingress の
  Host を一致させる（不一致で 404/503）。
- idle→0 の実時間 ≒ `requestRate.window` + `scaledownPeriod`。既定 1m+60s だと ~120s かかったので、
  cold-start を速く回すため window=30s / scaledownPeriod=15s に調整（idle→0 ≒ 45s）。

## 実測データ（背中合わせ A/B）
| 指標 | always-on (min=2) | scale-to-zero (min=0) |
|---|---|---|
| idle app Pod | **2** | **0** |
| idle app メモリ | 126Mi | 0（app） |
| cold-start avg / p95 | 41ms / 47ms | **4,730ms / 5,350ms** |
| burst 0→N エラー率 | 0% | **0%**（interceptor hold） |
| warm throughput / p95 | 82.6 req/s / 25.5ms | 82.6 req/s / 27.6ms |
| 常駐 制御面（共有固定費） | — | KEDA ~160Mi+ |

## いちばんの気づき — scale-to-zero は「idle コスト → cold-start latency」への変換
1. **cold-start 税 ≈ 115×**（4,730ms vs 41ms）。だが **warm 定常は完全一致**（82.6 req/s・p95 ~26ms・503 ゼロが
   両アーム）。**scale-to-zero は定常性能を一切犠牲にしない。払うのは idle 後の初回一発だけ**。壁は消えず
   「idle 常駐コスト」→「初回 latency」へ移動した（Lv5「壁は移動する」/ Lv8「latency 切り離し≠throughput増」の系譜）。
2. **cold-start はエラーでなく latency に出る**。interceptor が 0→1 の間ホールドするので burst でも 503 ゼロ
   （18,998 req 全部 200）。Lv2 の「probe 事故＝502 高速返しで throughput が嘘をつく」とは逆に、遅さが正直に p95 へ出る。
3. **scale-to-zero はタダではない（正直な控除）**。app Pod=0 でも KEDA 制御面（interceptor/scaler/operator/apiserver）が
   ~160Mi 常駐し、**1 workload なら always-on の 2 Pod(126Mi)より高くつく**。idle 節約が黒字化するのは
   「制御面固定費 ÷ workload あたり節約」の**損益分岐点を超える workload 数**から。=散発アクセスの休眠 workload が
   多数あるほど効く。単一・常時トラフィックなら always-on(min≥1)が正解。教訓6「数字は控除まで見ろ」の再演。

## 払ったコスト / トレードオフ
- **初回 latency**（cold-start 4.7s）。UX 直撃。production では最小 warm プール（min=1）や事前ウォームで緩和するが、
  それは「idle コストを一部払い戻す」＝ 取引の逆方向。
- **制御面の常駐固定費**（~160Mi + 運用複雑性）。beta の http-add-on 依存。production の HTTP scale-to-zero は Knative が定石。

## 実装メモ / 罠
- **Docker-Mac VM 歪み**: cold-start 4.7s には VM scheduler ジッタ + `app/src/server.ts` の起動処理
  （`CREATE TABLE` リトライ + テーブル可視待ち + DB 接続確立）が丸ごと乗る。測っているのは
  「k8s 起動 latency + アプリ起動 latency」の合算。実機/EKS ではもっと小さい。**相対差で読む**。
- interceptor/HTTPScaledObject のスキーマ・Service 名は chart バージョン依存（最頻の詰まりどころ）。
- `HTTPScaledObject.hosts` と Ingress の Host ヘッダ不一致 → 404/503。
- KEDA は裏で HPA(min=1) を作り、0↔1 は add-on の external-scaler、1↔N は HPA が担う。`ScalingDisabled:
  replica count is zero` は正常（0 のとき HPA は無効化される）。

## 次へ（Lv13 以降 / 別軸）
- トポロジ移行軸はこれで一区切り（compose→k8s→HPA→scale-to-zero）。次は **アーキ分解軸**：
  monolith ↔ modular monolith ↔ microservices。現状 app は単一 items ドメインで薄く切り出す相手がいないので、
  **items+orders+users の多ドメイン app を先に構築**してから（order→item 在庫参照 / user 認証の cross-context 依存）
  latency/障害伝搬/デプロイ単位を背中合わせ A/B で測る。
