# Lv23 — choreography saga：制御を中央から散らすと、結合は減るが「フローがどこにも無い」税を払う

記録日: 2026-07-15 / マシン: 8 コア Mac / Docker Compose(2DB・双方向) / 機能テスト

## やったこと
- Lv19 の saga は **orchestration**: `createOrderSaga()` 1 関数に reserve→insert→confirm/compensate の全フローがある(中央指揮・追跡可能)。
- Lv23 で **choreography** を追加(`SAGA_STYLE=orchestration(既定)|choreography`・orchestration は完全不変)。中央指揮者を消し、
  **各サービスがイベントを発行し合って**状態遷移する。Lv22 の outbox/inbox を**両サービスに**置き、双方向にした:
  1. orders: `POST /orders` → **同一 orders-db tx** で `order(status='pending')` + `OrderCreated` を choreo_outbox に emit。201(pending)を返す。
  2. orders poller → items `/internal/choreo/order-events` へ `OrderCreated` 配送。
  3. items: **同一 items-db tx** で dedup → 在庫チェック → 減算成功なら `StockReserved`・不足なら `StockRejected` を emit(補償の起点)。
  4. items poller → **orders `/internal/choreo/stock-events`** へ配送(★ items が orders にコールバック＝**双方向結合**)。
  5. orders: dedup → `status` を pending→`confirmed`(Reserved) / `cancelled`(Rejected)。
- **フローがどの単一関数にも無い** —— 2 サービスのイベントハンドラ + 2 poller + 2 outbox に散る。これが choreography の定義的性質。

## コマンドと出力
```console
### DEMO① happy（OrderCreated → StockReserved → confirmed）###
POST /orders {itemId:3, qty:5} → 201  {status:"pending"}          # 即時 pending・stock 未変化
  ~イベント: order.status=confirmed  stock=999995
  orders choreo_outbox: OrderCreated=delivered   items choreo_outbox: StockReserved=delivered

### DEMO② reject/補償（在庫不足 → StockRejected → cancelled）###
POST /orders {itemId:3, qty:2000000}(>在庫) → 201  {status:"pending"}
  ~イベント: order.status=cancelled  stock=999995(不変)            # 減算せず reject→cancel＝補償
  items choreo_outbox: StockReserved, StockRejected

### フローの所在（orchestration=1関数 vs choreography=散在）###
choreography の 1 注文の物語が跨るファイル: server.ts / orders{repo,ports,routes,usecase} / items{repo,ports,routes}  = 8 箇所
orchestration(createOrderSaga) の物語: create-order.ts の 1 関数
```

## orchestration との対比
| | orchestration（Lv19） | choreography（Lv23） |
|---|---|---|
| 制御の所在 | 中央 coordinator が全ステップを指揮 | 無い。各サービスがイベントに反応 |
| フローの可読性 | **1 関数** `createOrderSaga()` で読める | **8 ファイルに散る**。単一の場所に無い(emergent) |
| サービス間結合(topology) | 片方向(coordinator→items のみ) | **双方向**(items も orders にコールバック＝`ORDERS_EVENTS_URL`) |
| 「この saga の今の状態は?」 | coordinator の saga_log を見れば分かる | 複数サービスの outbox/inbox/status を突合しないと分からない |
| 中央のボトルネック/単一障害点 | coordinator に集中 | 無い(各自が進める) |
| デバッグ | ステップを 1 箇所で追える | イベントを跨いで追う(観測性の税) |

## わかったこと
1. **orchestration ↔ choreography は「制御の所在」の取引**。中央に集める(可読・変更容易・でも coordinator が全員を知る/ボトルネック)か、
   散らす(疎結合・ボトルネック無し・でもフローが emergent で「どこにも書いてない」)か。**複雑さを消すのでなく置き場所を選ぶ**
   (教訓33 の再演)。多くのチームが choreography の**観測性・デバッグの税**を過小評価する —— コードを読んでも 1 注文が何をするか分からない。
2. **choreography は結合を「関数呼び出し」から「トポロジ」へ移す**。orchestration は片方向(coordinator→participant)だが、
   choreography は participant も発行元にコールバックするので **everyone-talks-to-everyone** に近づく(`ORDERS_EVENTS_URL` の出現が象徴)。
   「疎結合」と言うが、消えたのはコンパイル時結合で、**実行時の相互依存(誰がどのイベントを待つか)は増える**。
3. **実際に踏んだ罠(qa 指摘・修正済)**: orders と items の受信 endpoint を両方 `POST /internal/choreo/events` にしたら、
   `SERVICE=all`(monolith)では両ドメインが**同一 Fastify インスタンス**に載るため `FST_ERR_DUPLICATED_ROUTE` で
   **Lv13-23 の全 hex 段が起動不能**になった。→ パスを `order-events`/`stock-events` に分離して解決。
   **choreography はイベント endpoint が増殖し、名前空間を慎重に切らないと衝突する** —— これ自体が「散らすことの管理コスト」の実例。

## 引っ越し軸 → 本番グレード化軸の締め（Lv16-23）
- Lv16-19 で「泥団子→解きほぐし→live 抽出→分散tx(none/2pc/saga)」を通し、Lv20-23 で分散tx を本番グレードへ:
  **実ネット税(裾)→ 2PC 自動回復(commit point)→ outbox(exactly-once 効果)→ choreography(制御の所在)**。
- 一貫する学び: **分散は「原子性・可用性・可読性」のどれかを必ず別の場所で支払う**。2PC はブロッキングで、saga は中間状態で、
  outbox は非同期で、choreography は観測性で。タダのものは無い —— 選ぶのは「壁の位置」だけ(教訓38 の総括)。
